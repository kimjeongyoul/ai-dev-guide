# [인프라 6편] 트래픽 제어 전략 — Rate Limiting 알고리즘 심층 분석

&nbsp;

고가용성 시스템에서 **Rate Limiting(처리율 제한)**은 단순히 요청을 막는 것이 아니라, 시스템의 가용 자원을 보호하고 특정 사용자의 자원 독점을 방지하여 전체적인 서비스 안정성을 유지하는 핵심 장치입니다. API Gateway 레이어에서 적용할 수 있는 주요 알고리즘의 작동 원리와 실무적 선택 기준을 분석합니다.

&nbsp;

---

&nbsp;

## 1. Token Bucket (토큰 버킷)

가장 유연하고 널리 사용되는 알고리즘입니다.

- **작동 원리**:
  1. 정해진 비율로 토큰이 버킷에 채워집니다 (최대 용량 존재).
  2. 요청이 들어오면 버킷에서 토큰을 하나 꺼내고 요청을 처리합니다.
  3. 토큰이 없으면 해당 요청은 버려지거나(429 Too Many Requests) 대기열에 추가됩니다.
- **장점**: **일시적인 폭증(Burst Traffic)을 허용**합니다. 버킷에 토큰이 쌓여 있다면, 짧은 시간에 몰려오는 요청들을 즉시 처리할 수 있어 사용자 경험 측면에서 유리합니다.
- **실무 적용**: 일반적인 REST API 서비스의 기본 전략으로 가장 권장됩니다. (예: AWS API Gateway, Nginx의 기본 방식)

&nbsp;

## 2. Leaky Bucket (유출 버킷)

트래픽의 흐름을 일정하게 유지하는 데 특화되어 있습니다.

- **작동 원리**:
  1. 요청이 들어오면 버킷(큐)에 담깁니다.
  2. 버킷 바닥의 구멍을 통해 요청이 **일정한 속도**로만 빠져나가 백엔드로 전달됩니다.
  3. 버킷이 꽉 차면 새로 들어오는 요청은 버려집니다.
- **장점**: 유입 속도와 상관없이 백엔드로 전달되는 트래픽이 매우 균일합니다(Traffic Smoothing). 백엔드 시스템의 처리 능력이 고정되어 있거나, 스파이크에 매우 취약한 경우에 적합합니다.
- **실무 적용**: 데이터베이스 쓰기 작업이나 외부 서드파티 API 호출 제한 등 엄격한 속도 제어가 필요한 경우에 사용합니다.

&nbsp;

## 3. Fixed Window Counter (고정 윈도우 카운터)

시간 단위를 고정하여 관리하는 가장 단순한 방식입니다.

- **작동 원리**: '1분당 100회'와 같이 고정된 시간 구간 내의 요청 횟수를 카운트합니다.
- **치명적 단점 (Boundary Problem)**: 구간의 경계 지점(예: 59초와 01초 사이)에 트래픽이 몰리면, 설정된 제한의 최대 2배에 달하는 요청이 한꺼번에 처리될 수 있습니다.
- **실무 적용**: 정확한 제한보다는 대략적인 할당량(Quota) 관리가 필요할 때 사용합니다.

&nbsp;

## 4. Sliding Window Log/Counter (슬라이딩 윈도우)

고정 윈도우의 경계 문제를 해결하기 위해 고안되었습니다.

- **작동 원리**: 고정된 구간이 아니라, **현재 시점부터 과거 1분간**의 로그나 카운트를 계산합니다.
- **장점**: 어느 시점에서 보더라도 정확한 속도 제한을 보장합니다.
- **단점**: 로그 방식은 메모리 소모가 크고, 카운터 방식은 구현이 복잡합니다.
- **실무 적용**: 보안이 중요하거나 트래픽 스파이크에 매우 민감한 고성능 시스템에서 사용합니다.

## 5. 시니어의 선택 기준 (Decision Matrix)

어떤 알고리즘을 선택할 것인가는 **"무엇을 보호하고 싶은가"**에 달려 있습니다.

| 기준 | 추천 알고리즘 | 이유 |
|------|--------------|------|
| **사용자 경험 중시** | **Token Bucket** | 일시적 몰림을 허용하여 응답 지연을 최소화함 |
| **백엔드 보호가 최우선** | **Leaky Bucket** | 백엔드에 전달되는 부하를 완벽하게 일정하게 유지함 |
| **단순 할당량 관리** | **Fixed Window** | 구현이 쉽고 리소스 소모가 적음 |
| **정밀한 제어 필요** | **Sliding Window** | 경계 지점의 스파이크까지 완벽히 차단함 |
