# [인프라 1편] Multi-Region Active-Active 아키텍처 — 글로벌 서비스의 가용성 극대화와 데이터 정합성 관리 전략

멀티리전, ActiveActive, 트래픽제어, 데이터정합성, GSLB, 고가용성, 인프라설계

&nbsp;

글로벌 서비스의 가용성을 극대화하기 위해 다중 리전(Multi-Region)에 Active-Active 구성을 설계하는 것은 필수적이지만, 이는 지연 시간과 데이터 일관성 사이의 트레이드오프를 수반합니다. 
 본 아티클에서는 리전 간 트래픽 제어 전략과 분산 환경에서의 데이터 동기화 메커니즘을 분석합니다.

---

## 1. 트래픽 유도 및 지연 시간 최적화

리전 간 거리는 물리적인 지연 시간을 발생시키므로, 사용자의 요청을 가장 효율적인 지점으로 유도하는 것이 최우선 과제입니다.

*   GSLB(Global Server Load Balancing): DNS 수준에서 사용자의 IP 주소를 기반으로 지연 시간이 가장 낮은 리전의 엔드포인트를 반환합니다. 최근에는 Anycast IP를 활용하여 네트워크 홉을 줄이고 장애 시 즉각적인 트래픽 절체(Failover)를 수행하는 방식이 선호됩니다.

---

## 2. 데이터 일관성 전략과 복제 메커니즘

CAP 이론에 따라, 네트워크 파티션 상황에서 가용성과 일관성 중 하나를 선택해야 합니다. 서비스의 특성에 따라 서로 다른 데이터 전략을 적용하는 설계가 필요합니다.

### 2.1 하드웨어 및 클라우드 네이티브 복제 활용
금융 결제와 같이 강한 일관성이 필요한 도메인에서는 AWS Aurora Global Database나 DynamoDB Global Tables와 같이 인프라 수준에서 관리되는 복제 솔루션을 활용합니다. 이들은 물리적인 스토리지 수준에서 비동기 복제를 수행하며, 리전 장애 시 데이터 손실을 최소화하면서 마스터 승격(Promotion)을 수행할 수 있는 기능을 제공합니다.

### 2.2 최종 일관성(Eventual Consistency) 확보
소셜 피드나 알림 서비스와 같이 즉각적인 정합성보다 가용성이 중요한 경우, 이벤트 기반 아키텍처를 통해 리전 간 데이터를 비동기적으로 전파합니다. 메시지 큐(Kafka, RabbitMQ)를 활용하여 각 리전의 독립성을 보장하되, 시간이 지남에 따라 모든 리전이 동일한 상태를 공유하도록 설계합니다.

---

## 3. 충돌 방지 및 해결 전략 (Conflict Resolution)

Active-Active 구성에서는 동일한 데이터에 대해 서로 다른 리전에서 동시에 수정이 발생할 수 있습니다.

*   Last Writer Wins (LWW): 가장 최근에 작성된 데이터를 최종 상태로 인정하는 방식입니다. 구현이 단순하지만, 미세한 시간 차이로 인해 중간 수정 사항이 유실될 수 있는 위험이 있습니다.
*   CRDT (Conflict-free Replicated Data Types): 데이터 구조 자체에 논리적인 타임스탬프나 카운터를 포함하여, 별도의 중재 과정 없이도 각 노드가 동일한 최종 상태로 수렴하도록 보장하는 알고리즘입니다. 분산 환경에서 정교한 정합성 관리가 필요할 때 효과적입니다.

결국 Multi-Region 설계는 인프라의 중복 구성을 넘어, 비즈니스 요건에 맞는 적절한 데이터 정합성 모델을 선택하고 구현하는 아키텍처적 의사결정 과정입니다.
