# [인프라 7편] 자원 최적화와 지능형 스케일링 — Right-sizing과 KEDA

&nbsp;

클라우드 비용 최적화는 단순히 '사양을 낮추는 것'이 아니라, **'워크로드에 가장 적합한 자원을 할당하고, 필요할 때만 사용하는 것'**입니다. 시니어 개발자가 가져야 할 리소스 라이트사이징(Right-sizing) 전략과 Kubernetes 환경에서의 지능형 스케일링 도구인 KEDA의 실무 적용법을 상세히 다룹니다.

&nbsp;

---

&nbsp;

## 1. 리소스 라이트사이징 (Right-sizing) 전략

비용 최적화는 성능과 안정성을 해치지 않는 범위 내에서 이루어져야 합니다.

### 1-1. 데이터 기반의 리소스 할당
- **95th Percentile 분석**: 평균 사용량은 의미가 없습니다. CPU/메모리 사용량의 95퍼센타일 데이터를 기준으로 파드의 Request/Limit을 설정하십시오. 이는 대부분의 스파이크를 수용하면서도 유휴 자원을 최소화하는 황금비율입니다.
- **VPA (Vertical Pod Autoscaler) 활용**: 수동으로 설정하는 대신, VPA를 `Recommendation` 모드로 운영하여 실제 워크로드에 기반한 최적의 리소스 가이드를 지속적으로 수집하고 적용합니다.

### 1-2. 인스턴스 패밀리 최적화
- **운영 환경 (Prod)**: 성능의 일관성이 중요하므로 **C 타입(연산 최적화)**이나 **M 타입(범용)**을 사용합니다. 예약 인스턴스(RI)나 세이빙 플랜을 통해 베이스라인 비용을 30~50% 절감합니다.
- **개발 환경 (Dev)**: 일시적인 부하가 잦으므로 **T 타입(Burstable)**을 적극 활용합니다. 업무 시간 외에는 인스턴스를 자동으로 종료하거나 파드 수를 0으로 줄이는 스케줄링을 도입합니다.

## 2. Kubernetes 스케일링 노하우: HPA

표준 HPA(Horizontal Pod Autoscaler)를 정교하게 사용하는 것만으로도 많은 문제를 해결할 수 있습니다.

### 2-1. 임계치 설정의 황금률
- **50~60% 설정**: 임계치를 80% 이상으로 잡으면 트래픽 급증 시 신규 파드가 준비되기 전에 기존 파드가 부하를 견디지 못하고 쓰러집니다. 50~60%는 시스템이 안전하게 확장될 수 있는 '대응 시간'을 벌어주는 수치입니다.
- **Cool-down (Stabilization)**: `stabilizationWindowSeconds` 설정을 통해 트래픽이 일시적으로 요동칠 때 파드가 빈번하게 생성/삭제되는 **Thrashing 현상**을 방지해야 합니다.

## 3. 지능형 스케일링의 핵심: KEDA

HPA가 '불이 난 뒤에 물을 뿌리는 것'이라면, KEDA(Kubernetes Event-driven Autoscaling)는 **'연기가 나기 시작할 때 미리 대처하는 것'**입니다.

### 3-1. 왜 KEDA인가?
표준 HPA는 CPU/메모리라는 사후 결과만 봅니다. 반면 KEDA는 외부 이벤트 소스를 직접 감시합니다.
- **Consumer Lag 기반**: Kafka나 RabbitMQ의 메시지가 쌓이는 속도(Lag)를 보고 파드를 늘립니다. CPU가 오르기 전에 이미 작업을 처리할 준비를 마칠 수 있습니다.
- **External Metrics**: DB의 특정 테이블 카운트, Redis의 큐 길이, 심지어 외부 API의 응답 결과에 따라 스케일링이 가능합니다.

### 3-2. Zero-scaling (Scale to Zero)
KEDA의 가장 큰 매력은 요청이 없을 때 자원을 0으로 줄일 수 있다는 점입니다.
- 이벤트가 발생하면 KEDA가 즉시 파드를 1개 이상으로 살려내어 요청을 처리하게 합니다. 이는 사내 배치 작업이나 간헐적으로 발생하는 워크로드의 비용을 극단적으로 줄여줍니다.

## 4. 시니어의 시각: 아키텍처적 결론

1.  **HPA**는 시스템 안정성을 위한 **최소한의 안전벨트**로 항상 켜두십시오.
2.  실질적인 비즈니스 대응은 **KEDA**를 통해 메시지 큐 적체량이나 이벤트 발생 여부를 기준으로 선제적으로 수행하십시오.
3.  주기적으로 **95th Percentile** 데이터를 검토하여 라이트사이징을 수행하는 조직적 문화를 만드십시오. 이것이 시니어 개발자가 증명해야 할 비용 효율의 실체입니다.
