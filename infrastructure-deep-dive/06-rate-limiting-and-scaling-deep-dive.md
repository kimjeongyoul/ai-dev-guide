# [인프라 6편] 트래픽 제어와 자원 최적화 — Rate Limiting 알고리즘과 KEDA 기반의 지능형 스케일링

&nbsp;

고가용성 시스템을 설계할 때 가장 중요한 두 가지 축은 **'어떻게 폭주하는 트래픽을 막아 시스템을 보호할 것인가'**와 **'어떻게 필요한 만큼만 자원을 사용하여 비용을 최적화할 것인가'**입니다. 시니어 엔지니어의 관점에서 API Gateway의 Rate Limiting 알고리즘과 Kubernetes의 지능형 스케일링 전략(Right-sizing, KEDA)을 심층적으로 분석합니다.

## 1. API Gateway Rate Limiting 알고리즘

시스템이 처리할 수 있는 한계를 넘어서는 요청이 들어올 때, 어떤 방식으로 이를 제한하느냐에 따라 사용자 경험과 시스템 안정성이 결정됩니다.

### 1-1. Token Bucket: 일시적 폭증(Burst) 허용
- **원리**: 일정한 속도로 토큰이 채워지는 바구니가 있고, 요청이 올 때마다 토큰을 하나씩 소비합니다. 바구니가 비어 있으면 요청을 거부합니다.
- **특징**: 바구니에 토큰이 쌓여 있다면 짧은 시간 내에 몰려오는 Burst 트래픽을 즉시 처리할 수 있습니다.
- **적합한 상황**: 일반적인 API 서비스. 사용자에게 어느 정도의 유연성을 제공하면서도 전체적인 평균 처리량을 제한하고 싶을 때 사용합니다. (가장 널리 쓰이는 방식)

### 1-2. Leaky Bucket: 균일한 흐름(Smoothing) 유지
- **원리**: 바구니 바닥에 구멍이 뚫려 있어 물(요청)이 일정한 속도로만 빠져나갑니다. 물이 넘치면(바구니가 꽉 차면) 버려집니다.
- **특징**: 유입 속도와 상관없이 출력 속도가 일정하므로 백엔드 서버에 전달되는 트래픽이 매우 균일합니다.
- **적합한 상황**: 백엔드 시스템의 처리 능력이 매우 엄격하게 고정되어 있어, 조금의 폭증도 허용할 수 없는 인프라 보호 위주의 환경.

### 1-3. Fixed / Sliding Window: 시간 단위 제한
- **Fixed Window**: 1분당 100회와 같이 고정된 시간 구간별로 카운트합니다. 구간 경계 지점에서 요청이 몰릴 경우(예: 59초에 100개, 1초에 100개), 실제로는 짧은 시간에 200개가 처리되는 '경계 문제(Boundary Problem)'가 발생합니다.
- **Sliding Window**: 고정된 구간이 아닌 현재 시점부터 과거 1분간의 요청을 계산합니다. 경계 문제를 해결하지만 구현이 복잡하고 메모리 사용량이 많습니다.

## 2. 클라우드 자원 최적화: Right-sizing 전략

Right-sizing은 단순히 서버 사양을 낮추는 것이 아니라, 워크로드의 특성에 가장 적합한 자원을 할당하는 엔지니어링 과정입니다.

### 2-1. 데이터 기반의 의사결정
- **95th Percentile 분석**: 평균(Average) 데이터는 스파이크를 반영하지 못합니다. CPU/메모리 사용량의 95퍼센타일 데이터를 기준으로 리소스의 Request/Limit을 설정하는 것이 안정성과 효율성의 균형점입니다.
- **Instance Family 최적화**: 단순히 CPU 개수만 맞추는 것이 아니라, 메모리 집약적 작업(M/R 타입), 연산 집약적 작업(C 타입), 혹은 일시적인 부하가 잦은 개발 환경(T 타입) 등 성격에 맞는 인스턴스 패밀리를 선택해야 합니다.

### 2-2. 운영 vs 개발 환경의 차별화
- **운영(Prod)**: C나 M 타입을 사용하며, 성능의 일관성을 위해 리소스 예약(Reserved Instances)이나 세이빙 플랜을 조합하여 비용을 절감합니다.
- **개발(Dev)**: T 타입(Burstable)을 적극 활용하고, 업무 시간 외에는 인스턴스를 자동으로 종료하는 스케줄링을 도입합니다.

## 3. Kubernetes 지능형 스케일링: HPA와 KEDA

표준 HPA는 CPU/메모리라는 '결과'를 보고 움직입니다. 하지만 진정한 고수는 부하가 발생하기 전 '징후'를 보고 움직입니다.

### 3-1. HPA 임계치 설정의 노하우
- **Golden Threshold**: 통상적으로 **50~60%**를 임계치로 설정합니다. 80% 이상은 트래픽 급증 시 신규 파드가 뜨기 전에 기존 파드가 OOM이나 뱅크럽시 상태에 빠질 위험이 크고, 30% 이하는 자원 낭비입니다.
- **Cooldown (Stabilization)**: 스케일 아웃 후 즉시 스케일 인이 발생하는 'Thrashing(요동)' 현상을 막기 위해 `stabilizationWindowSeconds`를 설정하여 시스템에 안정화 시간을 줍니다.

### 3-2. KEDA (Event-driven Autoscaling)의 실무 적용
KEDA는 CPU 대신 외부 이벤트(메시지 큐, DB 쿼리 결과, 외부 메트릭)를 기준으로 스케일링합니다.
- **Queue-based Scaling**: Kafka나 RabbitMQ의 **Consumer Lag**이 늘어나는 것을 보고 파드를 늘립니다. CPU가 오르기 전에 이미 작업을 처리할 일손을 늘리는 선제적 대응입니다.
- **Zero-scaling**: 요청이 전혀 없을 때는 파드를 0개로 줄였다가, 이벤트가 발생하면 즉시 1개 이상으로 살려내는 Serverless 구조를 K8s 상에서 구현할 수 있습니다.

## 4. 시니어의 시각: 아키텍처적 결론

1.  **사용자 경험 위주**라면 **Token Bucket**을 쓰십시오.
2.  **인프라 보호가 최우선**이라면 **Leaky Bucket**을 쓰십시오.
3.  **HPA**는 기본 안전장치로 두되, 실질적인 비즈니스 부하 대응은 **KEDA**와 같은 이벤트 기반 도구를 결합한 하이브리드 전략을 취하는 것이 현대적인 클라우드 아키텍처의 정석입니다.
