# [인프라 9편] 플랫폼 공학(Platform Engineering) — 개발팀의 자율성과 인프라 거버넌스의 균형을 어떻게 잡을 것인가?

&nbsp;

"내 애플리케이션을 띄우는 데 왜 일주일이나 걸리죠?" 마이크로서비스가 수백 개로 늘어난 조직에서 흔히 들리는 개발자의 비명이다. 과거의 인프라 팀은 티켓 시스템 뒤에 숨어 수동으로 리소스를 할당했지만, 이는 현대의 빠른 배포 속도를 감당할 수 없다. 하지만 개발자에게 클라우드 관리자 권한을 통째로 넘겨주는 순간, 인프라는 보안 위협과 비용 폭탄의 지옥으로 변한다. 

&nbsp;

이 절망적인 트레이드오프를 해결하기 위해 등장한 것이 바로 **IDP(Internal Developer Platform)**다. 본 글에서는 개발자에게는 자유를, 조직에게는 통제권을 부여하는 **플랫폼 공학**의 핵심 설계 전략을 파헤친다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. Golden Path: 규제된 자유의 설계

&nbsp;

IDP는 개발자에게 백지 상태의 인프라를 주는 것이 아니다. 대신, 조직이 검증한 **'황금길(Golden Path)'**을 깔아주는 것이다. 

&nbsp;

- **Curb Cuts (연석 경사로) 전략**: 휠체어를 위해 만든 경사로가 유모차와 여행 가방을 든 사람 모두에게 편안함을 주듯, 인프라 팀이 설계한 표준 템플릿(Terraform, Helm)은 초보 개발자에게는 가이드가 되고 숙련된 개발자에게는 지루한 수동 작업을 줄여주는 도구가 된다.
- **Self-Service Portal**: Backstage와 같은 도구를 활용해 개발자가 클릭 몇 번으로 데이터베이스를 생성하고 도메인을 할당받게 한다. 이때 배후에서는 인프라 팀이 설정한 보안 정책이 자동으로 주입된다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 가드레일: Policy as Code (PaC)를 통한 자동 거버넌스

&nbsp;

자유의 대가는 엄격한 감시다. 사람이 아닌 **코드(Policy as Code)**가 이를 수행해야 한다.

&nbsp;

- **OPA(Open Policy Agent)의 통합**: CI/CD 파이프라인 마지막 단계에서 "퍼블릭 서브넷 노출이 포함된 배포인가?", "비용이 지나치게 높은 인스턴스인가?"를 자동으로 검사하여 정책에 위반되는 배포를 원천 차단한다.
- **드리프트(Drift) 탐지**: 수동으로 변경된 인프라 설정을 실시간으로 감지하고, 플랫폼이 정의한 원래 상태로 되돌림으로써 '인프라 부채'의 발생을 막는다.

&nbsp;

&nbsp;

&nbsp;

# 3. 실전 엔지니어링 교훈: 플랫폼은 운영이 아니라 '제품'이다

&nbsp;

실패하는 플랫폼 공학팀은 자신들을 '고급 운영팀'으로 생각한다. 성공하는 팀은 자신들을 **'사내 개발자를 고객으로 삼는 프로덕트 팀'**으로 정의한다. 

1. 개발자의 **인지 부하(Cognitive Load)**를 줄이는 데 집중하라. 개발자가 쿠버네티스 YAML의 세부 명세를 공부하게 만드는 순간 플랫폼은 실패한 것이다.
2. 플랫폼은 점진적으로 확장하라. 모든 기능을 한꺼번에 자동화하려 하지 말고, 팀에서 가장 반복적이고 고통스러운 작업(예: DB 생성, 인증서 갱신)부터 해결해 나가는 것이 핵심이다.

IDP 아키텍처는 기술적 구현보다 조직의 **엔지니어링 문화와 프로세스**를 코드로 녹여내는 고도의 철학적 작업이다.
