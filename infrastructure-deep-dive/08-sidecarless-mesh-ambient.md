# [인프라 8편] Sidecarless 아키텍처 — 왜 서비스 메시는 Sidecar를 버리고 Ambient Mesh로 진화하는가?

&nbsp;

쿠버네티스(Kubernetes) 환경에서 서비스 메시(Service Mesh)는 마이크로서비스 간의 통신을 제어하는 강력한 표준으로 자리 잡았다. 하지만 모든 Pod 옆에 거대한 Envoy 프록시를 띄우는 **Sidecar 방식**은 머지 않아 '성능의 재앙'이 되었다. 수천 개의 컨테이너가 띄워질 때마다 기하급수적으로 늘어나는 메모리 점유율과 프록시 레이어를 지날 때마다 더해지는 네트워크 홉(Hop)은 엔지니어들에게 "배보다 배꼽이 더 크다"는 회의감을 안겨주었다. 

&nbsp;

본 글에서는 이러한 Sidecar의 물리적 한계를 돌파하기 위해 등장한 **Sidecarless(Ambient Mesh)** 아키텍처의 혁신적 설계 원리와 운영 효율성을 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. Sidecar Tax: 기존 방식이 지불해야 했던 대가

&nbsp;

기존의 Istio 모델은 모든 애플리케이션 Pod마다 별도의 프록시를 배치한다. 이는 논리적으로는 완벽한 격리를 제공하지만, 실전에서는 다음과 같은 치명적인 비용(Sidecar Tax)을 발생시켰다.

&nbsp;

- **리소스 낭비**: 프록시 하나가 수십 MB의 메모리만 차지해도, 수천 개의 Pod가 운영되는 환경에서는 인프라 전체 리소스의 20% 이상이 오직 '통신 보조'에만 쓰이게 된다.
- **업데이트의 공포**: 보안 취약점 해결을 위해 프록시를 업데이트하려면 모든 애플리케이션 Pod를 재시작해야 한다. 이는 애플리케이션 팀과 인프라 팀 간의 불필요한 마찰을 유발한다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. Ambient Mesh: 계층화된 프록시의 마법

&nbsp;

Ambient Mesh는 기능을 두 개의 레이어로 쪼개어 Sidecar 없이 메시를 구현한다.

&nbsp;

## 2-1. L4 보안 전송 레이어 (ztunnel)
각 노드(Node)마다 단 하나의 **ztunnel** 프록시를 배치한다. 이는 노드 내부의 모든 Pod 트래픽을 가로채서 mTLS(상호 인증 TLS)로 암호화하고 전송하는 역할만 수행한다. Pod를 수정하거나 재시작할 필요가 없는 **'공유 인프라'** 모델이다.

## 2-2. L7 정책 레이어 (Waypoint Proxy)
HTTP 라우팅이나 서킷 브레이커처럼 복잡한 처리가 필요한 서비스에 대해서만 별도의 **Waypoint 프록시**를 띄운다. 모든 트래픽이 아닌, '필요한 서비스'만 선택적으로 레이어 7 기능을 사용하게 함으로써 전체 시스템의 부하를 획기적으로 낮춘다.

&nbsp;

&nbsp;

&nbsp;

# 3. 실전 엔지니어링 교훈: 투명성이 운영의 핵심이다

&nbsp;

Sidecarless 아키텍처가 주는 가장 큰 선물은 **'애플리케이션 개발자와의 작별'**이다. 개발자는 자신의 애플리케이션 옆에 프록시가 있는지 없는지 알 필요가 없으며, 인프라 팀은 서비스 중단 없이 메시 레이어를 업데이트할 수 있다. 

&nbsp;

시니어 설계자라면 기술을 도입할 때 단순히 기능(Feature)만 보지 말고, 그 기술이 조직의 **운영 경계(Operational Boundary)**를 얼마나 깔끔하게 나누어 주는지 고민해야 한다. Ambient Mesh는 단순한 성능 최적화를 넘어, 인프라와 애플리케이션의 결합도를 완전히 끊어낸 클라우드 네이티브의 정점이다.
