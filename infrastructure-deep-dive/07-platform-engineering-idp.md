# [인프라 07편] 플랫폼 공학(Platform Engineering) — 개발팀의 자율성과 인프라 거버넌스의 균형을 어떻게 잡을 것인가?

IDP, 플랫폼공학, GoldenPath, IaC, 거버넌스, 셀프서비스, 인프라설계

&nbsp;

현대적인 개발 조직에서 인프라 팀은 병목이 되어서는 안 됩니다. 개발자가 직접 리소스를 생성할 수 있는 **IDP(Internal Developer Platform)** 구축은 필수적이지만, 무분별한 생성을 막기 위한 가드레일 설계가 그보다 더 중요합니다.

&nbsp;

---

&nbsp;

# 1. Golden Path: 검증된 길을 제공하라

&nbsp;

IDP의 핵심은 개발자에게 백지 상태의 인프라 권한을 주는 것이 아니라, 성공이 보장된 **'골든 패스'**를 깔아주는 것입니다.
- **표준화된 템플릿**: 보안과 비용 최적화가 완료된 Terraform Module이나 Helm Chart를 제공합니다.
- **셀프 서비스 UI**: 복잡한 클라우드 콘솔 대신, 조직의 정책이 반영된 전용 포털(Backstage 등)을 통해 리소스를 요청합니다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 가드레일: Policy as Code (PaC)의 도입

&nbsp;

자유를 주되, 실수는 시스템이 차단해야 합니다.
- **자동 검증**: OPA(Open Policy Agent)를 CI/CD에 통합하여 퍼블릭 서브넷 노출이나 비용이 비싼 인스턴스 생성을 자동으로 거부합니다.
- **수명 주기 관리**: 태그가 없는 리소스를 감지하고, 일정 기간 사용되지 않는 스테이징 자원은 자동으로 삭제하는 프로세스를 구축합니다.

&nbsp;

&nbsp;

---

&nbsp;

# 3. 실전 엔지니어링 교훈: 플랫폼은 제품이다

&nbsp;

성공적인 IDP는 인프라 팀의 편의가 아닌 **'개발자 경험(DX)'**에 초점을 맞춰야 합니다. 인프라 팀은 이제 단순 운영자가 아니라, 내부 고객(개발자)에게 인프라라는 프로덕트를 제공하는 플랫폼 엔지니어로 거듭나야 합니다.
