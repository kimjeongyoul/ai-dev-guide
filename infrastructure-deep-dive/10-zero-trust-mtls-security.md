# [인프라 10편] Zero Trust 보안 — 서비스 메시와 mTLS를 통해 내부 침입으로부터 시스템을 어떻게 보호할 것인가?

ZeroTrust, mTLS, 서비스메시, 인증인가, Identity보안, 네트워크보안, 인프라설계

&nbsp;

전통적인 네트워크 보안은 외부 성벽을 쌓는 방식이었습니다. 하지만 성벽이 뚫리면 내부의 모든 데이터는 무방비로 노출됩니다. 이제는 모든 통신을 의심하고 검증하는 **Zero Trust 아키텍처**가 표준입니다.

&nbsp;

---

&nbsp;

# 1. Identity 중심의 보안 설계 (SPIFFE)

&nbsp;

IP 기반 방화벽은 컨테이너 환경에서 무용지물입니다.
- **Identity 부여**: 서비스마다 고유한 ID(Identity)를 발급하고, 이를 기반으로 인증을 수행합니다.
- **비침습적 적용**: 애플리케이션 코드 수정 없이 서비스 메시 레이어에서 자동으로 ID를 관리하고 배포합니다.

# 2. mTLS와 세밀한 접근 제어 (RBAC)

&nbsp;

## 2-1. 상호 TLS (mTLS) 인증
서비스 간 통신 시 클라이언트와 서버가 서로를 인증합니다. 이를 통해 네트워크 상의 패킷 감청을 방지하고, 허가받지 않은 서비스가 내부 API를 호출하는 것을 원천 차단합니다.

## 2-2. 계층적 권한 검증
연결이 허용되었다고 해서 모든 데이터에 접근할 수 있는 것은 아닙니다. 사용자 JWT와 서비스 권한을 결합하여, 특정 서비스가 특정 엔드포인트를 호출할 수 있는지 **화이트리스트 기반**으로 엄격히 관리합니다.

# 3. 실전 엔지니어링 교훈: 관찰성이 보안이다

&nbsp;

Zero Trust의 완성은 모니터링입니다. 모든 통신 기록을 로깅하고, 평소와 다른 패턴(갑작스러운 대용량 데이터 전송 등)이 감지되면 즉시 해당 서비스의 Identity를 무효화하여 격리해야 합니다. 보안은 고정된 설정이 아니라 **지속적으로 움직이는 프로세스**입니다.
