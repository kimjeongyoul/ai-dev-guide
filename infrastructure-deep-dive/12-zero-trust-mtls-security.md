# [인프라 12편] Zero Trust 보안 — 서비스 메시와 mTLS를 통해 내부 침입으로부터 시스템을 어떻게 보호할 것인가?

&nbsp;

과거의 보안 아키텍처는 거대한 성벽을 쌓는 것과 같았다. 강력한 방화벽(WAF)을 세우고 그 안에 모든 서비스를 가두어 두면 안전하다고 믿었다. 하지만 이 '경계 보안' 모델은 한 번 성벽이 뚫리는 순간, 내부의 모든 데이터가 해커의 놀이터가 되는 치명적인 결함을 가지고 있었다. 현대의 클라우드 네이티브 환경에서는 더 이상 내부망이라는 이유만으로 신뢰해서는 안 된다. 

&nbsp;

**"아무도 믿지 마라, 모든 것을 검증하라(Never Trust, Always Verify)"**는 **Zero Trust** 철학이 인프라 보안의 새로운 표준이 된 이유다. 본 글에서는 서비스 메시와 mTLS를 활용해 내부 침입으로부터 시스템을 철통같이 방어하는 구체적인 아키텍처를 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. Identity 중심 보안: IP 기반 신뢰의 종말

&nbsp;

컨테이너 환경에서 Pod의 IP는 수시로 변한다. 어제는 신뢰할 수 있었던 IP가 오늘은 해커가 장악한 컨테이너의 IP일 수 있다. 

&nbsp;

- **SPIFFE/SPIRE 기반의 Identity**: IP가 아닌 서비스의 고유한 정체성(Identity)을 기반으로 인증을 수행한다. 각 서비스는 자신의 정체성을 증명할 수 있는 암호화된 토큰을 발급받으며, 이는 서비스 간 통신의 주민등록증 역할을 한다.
- **동적 워크로드 인증**: 서비스가 배포되는 순간, 보안 에이전트는 해당 워크로드의 이미지 해시값, 네임스페이스 등을 검증하여 올바른 Identity를 부여한다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. mTLS: 통로를 지나는 모든 패킷을 암호화하라

&nbsp;

단순한 TLS가 서버만 인증한다면, **mTLS(Mutual TLS)**는 클라이언트와 서버가 서로의 신원을 상호 확인한다.

&nbsp;

- **상호 인증 및 암호화**: 서비스 간에 오가는 모든 패킷은 암호화되어 네트워크 상에서 탈취되더라도 내용을 알 수 없다. 또한, 허가받지 않은 서비스가 내부 API에 접근하려고 하면 인증 단계에서 즉시 차단된다.
- **사이드카(Sidecar)의 역할**: 애플리케이션 소스 코드에 복잡한 암호화 로직을 넣을 필요가 없다. Istio나 Linkerd 같은 서비스 메시의 프록시 레이어에서 mTLS 설정을 자동으로 관리하고 인증서를 주기적으로 갱신해 준다.

&nbsp;

&nbsp;

&nbsp;

# 3. 실전 엔지니어링 교훈: 관찰성이 곧 보안이다

&nbsp;

Zero Trust의 완성은 완벽한 차단이 아니라 **'철저한 감시'**에 있다.

1. **세밀한 RBAC(권한 제어)**: "A 서비스는 B 서비스의 `/health` 경로만 호출할 수 있다"는 식의 최소 권한 원칙(Least Privilege)을 적용하라.
2. **이상 징후 탐지**: 모든 서비스 간 통신 기록을 로깅하고, 평소와 다른 패턴(예: 갑작스러운 대용량 트래픽 이동)이 감지되면 즉시 해당 서비스의 Identity를 무효화하고 격리(Isolation)하는 자동화된 대응 파이프라인을 구축해야 한다.

보안은 고정된 상태가 아니라 지속적으로 움직이는 프로세스다. Zero Trust는 기술을 넘어 조직 전체가 **"이미 우리 내부에 적이 있다"**는 전제하에 움직이는 가장 강력한 방어 전략이다.
