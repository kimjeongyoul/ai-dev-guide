# [인프라 08편] 멀티 클라우드 아키텍처 — 리전 간 지연 시간과 데이터 복제 비용 문제를 어떻게 최적화할 것인가?

&nbsp;

"한 바구니에 모든 달걀을 담지 마라." 클라우드 벤더의 대규모 장애를 겪어본 기업들에게 멀티 클라우드(Multi-Cloud)는 더 이상 선택이 아닌 필수 생존 전략이다. 하지만 AWS와 Azure, 혹은 GCP를 동시에 사용하는 순간, 엔지니어는 인프라 역사상 가장 가혹한 두 가지 적을 마주하게 된다. 바로 **'물리적 거리로 인한 지연 시간(Latency)'**과 **'클라우드 경계를 넘나드는 데이터 전송 비용(Egress Fee)'**이다. 

&nbsp;

본 글에서는 벤더 종속성(Lock-in) 탈피라는 명분을 유지하면서도, 기술적 효율성과 경제성을 모두 잡는 멀티 클라우드 네트워크 및 데이터 아키텍처 설계법을 다룬다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. 벤더 간 고속도로: Cloud Interconnect의 비밀

&nbsp;

서로 다른 클라우드 간의 통신을 퍼블릭 인터넷에 맡기는 것은 자살 행위와 다름없다. 보안은 취약하고 지연 시간은 요동치며, 비용은 통제 불능이 된다.

&nbsp;

- **Carrier-Neutral IDC 활용**: Equinix나 Digital Realty 같은 중립적 데이터센터에 다이렉트 커넥트 장비를 두고, 각 클라우드 벤더(AWS DX, Azure ExpressRoute)를 1:1로 직접 연결한다. 이를 통해 수십 밀리초(ms)에 달하던 리전 간 지연 시간을 한 자릿수로 단축할 수 있다.
- **Anycast IP와 GSLB**: 전 세계에 흩어진 사용자 요청을 가장 가까운 클라우드 리전으로 유도하고, 특정 리전 장애 시 즉시 타 클라우드로 트래픽을 절체하는 지능형 라우팅 레이어를 배치한다.

# 2. 데이터 중력(Data Gravity)과 요약 전송 전략

&nbsp;

데이터는 '중력'이 있다. 쌓이면 쌓일수록 옮기기 어려워지고, 주변의 연산 자원(Compute)을 자신에게 끌어당긴다. 

&nbsp;

- **지역적 연산(Local Compute)**: 모든 로우(Raw) 데이터를 타 클라우드로 복제하려 하지 마라. 대신 데이터가 발생하는 리전 내에서 1차 처리를 수행하고, 타 클라우드에는 비즈니스에 핵심적인 **요약 정보(Summary)**나 **집계된 메트릭**만 비동기적으로 전송한다.
- **분산 정합성 전략**: 강한 일관성이 필요한 결제 데이터는 인프라 레벨의 복제(DynamoDB Global Tables 등)를 활용하고, 사용자 프로필이나 소셜 피드 같은 데이터는 메시지 큐를 통한 **최종 일관성(Eventual Consistency)** 모델을 적용하여 리전 간 종속성을 끊어낸다.

# 3. 실전 엔지니어링 교훈: 복잡도의 세금(Tax)을 계산하라

&nbsp;

시니어 설계자는 "기술적으로 가능하다"는 말과 "비즈니스적으로 가치 있다"는 말을 구분해야 한다.

1. 멀티 클라우드는 단일 클라우드보다 최소 **3배 이상의 운영 복잡도**를 발생시킨다. 각 벤더의 서로 다른 보안 그룹 설정, IAM 권한, 네트워크 명세 등을 관리하는 것은 엔지니어링 리소스를 엄청나게 소모한다.
2. "벤더 종속 탈피"라는 목표가 실제 장애 발생 빈도와 그로 인한 손실액 대비 경제적인지 냉정하게 계산하라. 가용성을 위해 응답 속도를 2배 희생하는 설계는 멀티 클라우드의 함정이다.

결국 멀티 클라우드의 승자는 **'가장 많은 클라우드를 쓰는 자'가 아니라, '데이터 이동의 비용을 가장 영리하게 통제하는 자'**다.
