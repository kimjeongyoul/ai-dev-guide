# [인프라 3편] Kubernetes 스케일링 전략 — 대규모 트래픽 폭증에 선제적으로 대응하기 위한 기술적 도전

Kubernetes, 오토스케일링, HPA, 메트릭서버, 성능최적화, 인프라확장, 인프라설계

&nbsp;

Kubernetes 환경에서 트래픽 폭증에 대응하기 위해 흔히 HPA(Horizontal Pod Autoscaler)를 사용하지만, 초단위로 급증하는 이벤트 상황에서는 메트릭 수집 지연과 파드 기동 시간으로 인해 한계에 부딪힙니다. 
 본 아티클에서는 이를 해결하기 위한 선제적 확장 및 Warm-up 전략을 분석합니다.

---

## 1. 표준 HPA의 한계와 병목 분석

HPA는 일반적으로 다음과 같은 단계에서 지연이 발생합니다.

1.  메트릭 수집 주기 (Metrics Server): CPU/메모리 사용량이 임계치를 넘었음을 인지하는 데 시간이 걸립니다.
2.  이미지 풀링 및 컨테이너 시작: 노드에 이미지가 없을 경우 다운로드 시간이 소요되며, 애플리케이션 프레임워크가 구동되는 데 추가적인 시간이 필요합니다.
3.  Cluster Autoscaler 지연: 파드를 띄울 리소스가 부족하여 신규 노드를 추가할 때 수 분의 시간이 소요될 수 있습니다.

---

## 2. 대응 전략: 예측 및 이벤트 기반 스케일링

### 2.1 Keda (Kubernetes Event-driven Autoscaling) 활용
CPU나 메모리 사용량과 같은 사후 메트릭이 아닌, 외부 이벤트 소스를 기준으로 스케일링을 수행합니다. 예를 들어 Kafka의 컨슈머 랙(Lag)이나 메시지 큐의 잔량을 실시간으로 감시하여 부하가 실제 서버에 반영되기 전에 미리 파드를 확장합니다.

### 2.2 예측 기반 오토스케일링 (Predictive Autoscaling)
과거의 트래픽 패턴을 기계학습 모델로 분석하여, 특정 시간에 트래픽이 몰릴 것을 미리 예측하고 선제적으로 스케일 아웃을 수행하는 방식입니다.

---

## 3. 리소스 가용성 확보를 위한 Warm-up 아키텍처

트래픽 폭증 시 즉각적인 대응을 위해 인프라 수준에서 버퍼를 구축해야 합니다.

*   Overprovisioning (Pause Pods): 우선순위가 매우 낮은 'Pause Pod'을 클러스터에 미리 띄워놓습니다. 실제 워크로드 파드가 생성될 때 리소스가 부족하면 Kubernetes 스케줄러가 Pause Pod을 즉시 축출(Eviction)하고 그 자리에 실제 파드를 배치함으로써 노드 추가 시간을 벌 수 있습니다.
*   Proactive 노드 확보: 대규모 이벤트가 예정된 경우 Cluster Autoscaler의 최소 노드 수를 미리 상향 조정하여 인프라 준비 시간을 확보합니다.

---

## 4. 애플리케이션 시작 최적화

인프라의 확장 속도만큼 애플리케이션의 기동 속도도 중요합니다.

*   이미지 캐싱 가속화: 노드별로 자주 사용하는 이미지를 미리 캐싱하거나, 이미지 레이어를 최적화하여 풀링 시간을 단축합니다.
*   Startup Probe 활용: 애플리케이션이 완전히 준비될 때까지 트래픽이 유입되지 않도록 정교하게 관리하여 서비스 안정성을 확보합니다.

결국 대규모 트래픽 대응은 단순한 자동화 설정을 넘어, 인프라 리소스 확보와 애플리케이션의 기동 메커니즘이 유기적으로 결합된 시스템적 설계가 필요합니다.
