# [인프라 06편] Sidecarless 아키텍처 — 왜 서비스 메시는 Sidecar를 버리고 Ambient Mesh로 진화하는가?

&nbsp;

Kubernetes 생태계에서 서비스 메시(Service Mesh)는 필수적인 인프라 레이어로 자리 잡았습니다. 하지만 모든 Pod에 Envoy 프록시를 띄우는 Sidecar 방식은 막대한 비용과 운영 복잡도를 야기했죠. 이제 그 대안으로 떠오른 **Sidecarless(Ambient Mesh)**의 설계 원리를 분석합니다.

&nbsp;

---

&nbsp;

# 1. Sidecar Tax: 기존 방식의 한계

&nbsp;

수천 개의 마이크로서비스가 운영될 때, Sidecar 프록시가 점유하는 리소스는 결코 무시할 수 없는 수준이 됩니다.
- **리소스 낭비**: 각 Pod마다 프록시가 CPU와 메모리를 점유하여 전체 인프라 비용의 10~20%를 차지.
- **운영 경계**: 프록시 업데이트 시 애플리케이션 Pod를 재시작해야 하는 강한 결합 발생.

&nbsp;

&nbsp;

---

&nbsp;

# 2. Ambient Mesh의 계층적 분리 설계

&nbsp;

Sidecarless 모델은 메시의 기능을 두 가지 계층으로 분리하여 효율성을 극대화합니다.

## 2-1. ztunnel (Zero Trust Tunnel)
노드(Node)당 하나의 공유 프록시를 둡니다. 이는 오직 L4 영역의 보안 전송(mTLS)과 관찰성만을 담당하며, 애플리케이션의 수정 없이 투명하게 동작합니다.

## 2-2. Waypoint Proxy
HTTP 라우팅이나 서킷 브레이커 같은 복잡한 L7 정책이 필요한 경우에만 별도로 실행됩니다. 특정 서비스 단위로 동작하며, 필요한 서비스만 비용을 지불하고 고급 기능을 사용할 수 있게 합니다.

# 3. 실전 엔지니어링 교훈: 비용 효율적 가용성

&nbsp;

Sidecarless 아키텍처는 기술적 혁신을 넘어 **'비용 효율성'**이라는 비즈니스 가치를 제공합니다. 서비스 메시 도입을 주저하게 만들었던 리소스 오버헤드를 해결함으로써, 더 많은 조직이 Zero Trust 보안과 고도화된 트래픽 제어를 누릴 수 있게 되었습니다.
