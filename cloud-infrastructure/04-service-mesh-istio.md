# [플랫폼 4편] 서비스 메시(Service Mesh) — Istio의 Envoy 프록시가 사이드카로 통신을 제어하는 원리

&nbsp;

마이크로서비스 아키텍처(MSA)가 거대해질수록 서비스 간 통신의 복잡도는 지옥으로 변한다. "어느 서비스가 어느 서비스를 부르는가?", "통신이 암호화되어 있는가?", "특정 서비스가 느려졌을 때 전체 시스템으로 장애가 전파되는가?" 이러한 고민을 해결하기 위해 등장한 것이 **서비스 메시(Service Mesh)**다. 

&nbsp;

그중 업계 표준인 **Istio**는 애플리케이션 코드의 수정 없이 인프라 레이어에서 모든 통신을 가로채고 제어한다. 본 글에서는 Istio의 핵심인 **사이드카(Sidecar) 주입 메커니즘**과 **Envoy 프록시**의 내부 동작, 그리고 **mTLS(Mutual TLS)** 상호 인증이 실질적으로 어떻게 패킷 레벨에서 구현되는지 심층 분석한다.

&nbsp;

&nbsp;

&nbsp;

# 1. 사이드카 주입(Injection)의 실체: Mutating Webhook

&nbsp;

Istio는 어떻게 우리가 배포한 파드(Pod) 옆에 슬그머니 Envoy 프록시 컨테이너를 끼워 넣는 것일까? 이 마법의 정체는 쿠버네티스의 **Mutating Admission Webhook**에 있다.

&nbsp;

## 1-1. 배포 라이프사이클 가로채기
1. 사용자가 `kubectl apply -f deployment.yaml`을 실행한다.
2. K8s API 서버는 이 요청을 받자마자 Istio의 `istiod` 컨트롤 플레인에 "이 파드를 배포해도 될까?"라고 묻는다.
3. `istiod`는 해당 네임스페이스에 `istio-injection=enabled` 레이블이 있는지 확인한다.
4. 레이블이 있다면, `istiod`는 원본 YAML의 파드 명세에 `istio-proxy`라는 이름의 Envoy 컨테이너와 `istio-init`이라는 초기화 컨테이너 설정을 강제로 삽입(Mutation)하여 API 서버로 돌려준다.
5. 최종적으로 노드에는 우리가 명령하지 않은 프록시가 함께 뜬다.

&nbsp;

## 1-2. iptables와 트래픽 하이재킹
사이드카가 떴다고 해서 트래픽이 자동으로 프록시를 거치는 것은 아니다. `istio-init` 컨테이너가 실행되는 짧은 순간, 노드의 커널 네트워킹 규칙인 **iptables**를 수정한다.
- 모든 인바운드(Inbound)와 아웃바운드(Outbound) TCP 패킷을 15001/15006 포트(Envoy가 점유한 포트)로 리다이렉트 시킨다.
- 결과적으로 애플리케이션 프로세스는 자신이 직접 통신한다고 착각하지만, 실제로는 모든 패킷이 커널 단에서 Envoy 프록시로 강제 전달된다.

# 2. Envoy 프록시: 서비스 메시의 팔다리

&nbsp;

Envoy는 C++로 작성된 고성능 L7 프록시다. Istio의 모든 데이터 플레인(Data Plane) 로직은 Envoy 내부의 **필터 체인(Filter Chain)**에서 수행된다.

&nbsp;

- **리스너(Listeners)**: 포트를 열고 트래픽을 대기.
- **클러스터(Clusters)**: 트래픽을 보낼 목적지 서비스들의 집합.
- **xDS API**: `istiod`로부터 실시간으로 라우팅 규칙(VirtualService), 서킷 브레이커 설정 등을 내려받는 동적 설정 인터페이스.

&nbsp;

사용자가 Istio 설정을 변경하면 `istiod`는 xDS 프로토콜을 통해 수만 대의 Envoy 프록시에게 설정을 1초 이내에 전파한다. 이 동적 업데이트 능력이 Envoy를 다른 프록시(Nginx 등)와 차별화하는 핵심 요소다.

# 3. mTLS (Mutual TLS): 제로 트러스트 보안의 완성

&nbsp;

Istio를 쓰는 가장 큰 이유 중 하나는 **mTLS**다. 서비스 간의 통신에서 클라이언트와 서버가 서로의 인증서를 확인하고 통신 전체를 암호화한다.

&nbsp;

## 3-1. 인증서 발급과 로테이션
`istiod` 내부의 CA(Certificate Authority)는 각 서비스의 고유 신원인 **SPIFFE ID**가 담긴 인증서를 자동으로 발급한다. 이 인증서는 파드 내의 시크릿(Secret)으로 마운트되는 것이 아니라, Envoy와 `istiod` 간의 **SDS(Secret Discovery Service)** API를 통해 메모리 상에서만 교환된다. 

&nbsp;

## 3-2. 핸드쉐이크 과정
1. **Envoy A (클라이언트)**: 아웃바운드 패킷을 잡고, 자신의 인증서를 제시하며 Envoy B와 TLS 핸드쉐이크를 시작한다.
2. **Envoy B (서버)**: 클라이언트의 인증서가 신뢰할 수 있는 CA로부터 발급되었는지 확인하고 자신의 인증서를 돌려준다.
3. 양쪽이 서로를 확인하면 대칭키를 생성하여 실제 데이터(HTTP/gRPC)를 암호화한다.
4. **애플리케이션 프로세스**: 암호화가 일어나는지도 모른 채 평문으로 Envoy와 대화한다. (격리 성공)

# 4. 트레이드 오프: 성능과 복잡성

&nbsp;

서비스 메시는 공짜가 아니다.
- **지연 시간(Latency)**: 패킷이 커널과 프록시 사이를 왕복하면서 발생하는 오버헤드로 인해 홉당 1~2ms의 지연이 추가된다.
- **리소스 소모**: 수천 개의 파드가 떠 있다면 각 파드마다 50~100MB의 메모리를 점유하는 Envoy 프록시의 합계 비용은 무시할 수 없다.

&nbsp;

이를 해결하기 위해 최근에는 사이드카를 없애고 노드당 하나의 공유 프록시만 두는 **Istio Ambient Mesh** 아키텍처가 제안되고 있다.

# 결론: 인프라가 앱의 언어를 이해하다

&nbsp;

Istio는 단순한 통신 도구가 아니라, 인프라가 애플리케이션의 '의도'를 이해하게 만드는 지능형 레이어다. "이 유저는 VIP니까 응답 속도가 빠른 노드로 보내줘" 혹은 "이 서비스는 불안정하니까 3번 실패하면 10초간 통신을 차단해줘"라는 비즈니스 로직을 인프라 레벨에서 구현할 수 있게 된다. 

&nbsp;

사이드카의 동작 원리를 이해하는 엔지니어만이 복잡한 네트워크 장애 상황에서 패킷의 흐름을 추적하고, 성능과 보안 사이의 최적의 균형점을 찾아낼 수 있다.
