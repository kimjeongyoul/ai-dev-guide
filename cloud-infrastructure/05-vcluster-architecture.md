# [플랫폼 5편] 가상 클러스터(vCluster) — 하나의 K8s에서 수백 개의 독립된 클러스터를 운영하는 아키텍처

&nbsp;

쿠버네티스(Kubernetes)를 운영하다 보면 '멀티 테넌시(Multi-tenancy)'의 장벽에 부딪힌다. 수백 명의 개발자에게 하나의 클러스터를 공유하게 하자니 네임스페이스 격리만으로는 CRD(Custom Resource Definition) 충돌이나 API 서버 부하를 막을 수 없다. 그렇다고 개발 팀마다 물리적인 클러스터를 따로 구축해 주자니 클라우드 비용과 관리 포인트가 기하급수적으로 늘어난다.

&nbsp;

이 딜레마를 해결하기 위해 등장한 혁신이 바로 **vCluster (Virtual Cluster)**다. 호스트 클러스터 내부의 한 네임스페이스 안에서 '완벽히 독립된 가짜 K8s 클러스터'가 돌아가게 만드는 기술이다. 본 글에서는 vCluster의 핵심인 API 서버 가상화와 동기화 엔진(Syncer)의 내부 메커니즘을 심층 분석한다.

&nbsp;

&nbsp;

# 1. 왜 네임스페이스 격리만으로는 부족한가?

&nbsp;

K8s의 기본 격리 단위인 네임스페이스는 다음과 같은 한계를 갖는다.
- **전역 자원 공유**: CRD, 스토리지 클래스, 노드 정보 등은 클러스터 전역 자원이다. 테넌트 A가 설치한 특정 버전의 Operator가 테넌트 B의 환경을 망가뜨릴 수 있다.
- **RBAC의 복잡성**: 네임스페이스가 늘어날수록 권한 관리 지옥(RBAC Hell)이 펼쳐진다.
- **API 서버 병목**: 모든 테넌트의 요청이 하나의 API 서버로 몰리면서 컨트롤 플레인의 안정성이 저하된다.

&nbsp;

vCluster는 테넌트에게 **'독립된 API 서버와 데이터 저장소(etcd)'**를 통째로 쥐여줌으로써 이 문제를 해결한다.

&nbsp;

&nbsp;

&nbsp;

# 2. vCluster 아키텍처: 클러스터 안의 클러스터

&nbsp;

vCluster는 호스트 클러스터 입장에서는 단순히 하나의 네임스페이스에서 돌아가는 **스테이트풀셋(StatefulSet)** 파드에 불과하다. 하지만 그 파드 내부에는 K8s의 핵심 컴포넌트들이 압축되어 들어있다.

&nbsp;

## 2-1. 가상 컨트롤 플레인 (Virtual Control Plane)
vCluster 파드 내부에는 k3s나 k0s 같은 경량 K8s 배포판이 실행된다.
- **Virtual API Server**: 테넌트의 모든 `kubectl` 명령을 직접 수신한다. 테넌트는 자신이 실제 K8s 마스터와 통신한다고 믿게 된다.
- **가상 데이터 저장소**: SQLite나 가벼운 etcd를 사용하여 해당 가상 클러스터만의 자원 정보를 저장한다. 호스트 클러스터의 데이터와는 완전히 분리된다.

&nbsp;

## 2-2. 동기화 엔진 (The Syncer)
vCluster에는 실제 '노드(Node)'가 없다. 따라서 파드를 띄우려면 호스트 클러스터의 자원을 빌려야 한다. 이 가교 역할을 하는 것이 **Syncer**다.
1. 사용자가 vCluster API 서버에 "파드 A 생성" 명령을 내린다.
2. vCluster 내부 저장소에 정보가 기록된다.
3. **Syncer**가 이 이벤트를 감지하고, 호스트 클러스터의 특정 네임스페이스에 실제 파드 객체를 생성하도록 요청한다. 
4. 이때 파드 이름은 충돌 방지를 위해 `vcluster-name_pod-name_namespace` 형태로 변환(Mapping)된다.

# 3. 데이터 평면의 가상화와 매핑

&nbsp;

vCluster의 가장 놀라운 점은 '노드 가상화'다.

&nbsp;

- **노드 동기화**: vCluster 내부에서 `kubectl get nodes`를 치면 호스트 클러스터의 실제 노드 정보가 가공되어 나타난다. Syncer는 호스트 노드 중 일부를 선택하여 가상 클러스터 소유인 것처럼 속인다.
- **네트워크와 서비스**: vCluster 내부의 Service와 Ingress는 Syncer에 의해 호스트 클러스터의 실제 네트워크 객체와 1:1로 매핑된다. 따라서 가상 클러스터 간에도 필요하다면 서로 통신이 가능하다.

# 4. 실전 활용 시나리오: 비용과 개발 경험의 조화

&nbsp;

## 4-1. 일회용 개발 환경 (Ephemeral Clusters)
CI/CD 파이프라인에서 테스트를 돌릴 때마다 실제 클러스터를 프로비저닝하려면 10분 이상 소요된다. vCluster는 10초 이내에 생성이 완료되므로, 각 PR(Pull Request)마다 독립된 K8s 환경을 제공하고 테스트 종료 후 즉시 파기하는 아키텍처를 최소한의 비용으로 구현할 수 있다.

&nbsp;

## 4-2. Operator 및 CRD 테스트
시스템 전역을 오염시킬 위험이 있는 새로운 버전의 Operator나 복잡한 CRD를 메인 클러스터의 영향 없이 안전하게 검증할 수 있다. 가상 클러스터 내부에서 어떤 사고를 쳐도 호스트 클러스터는 평온함을 유지한다.

# 결론: 진정한 인프라 민주화

&nbsp;

vCluster는 플랫폼 엔지니어에게 "최소한의 비용으로 최대한의 격리를 제공하는" 궁극의 멀티 테넌시 도구다. 관리자는 하나의 거대한 클러스터만 관리하고, 개발 팀에게는 마치 각자가 클러스터 주인인 것 같은 환상을 심어줄 수 있다. 

&nbsp;

추상화 레이어가 하나 더 추가되는 만큼 네트워크 복잡도가 올라가지만, 이를 통해 얻는 운영 효율성과 비용 절감 효과는 그 단점을 상쇄하고도 남는다. 인프라는 이제 더 이상 고정된 물리 자원이 아니라, 필요에 따라 쪼개고 합치는 '논리적 소프트웨어'로 완전히 진화했다.
