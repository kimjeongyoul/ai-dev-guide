# [프런트엔드 08편] RSC와 상태 관리의 변화 — 서버 컴포넌트 도입 이후 클라이언트 전역 상태의 역할은 어떻게 재정의되는가?

&nbsp;

**React Server Components(RSC)**는 프런트엔드의 데이터 페칭과 상태 관리 패러다임을 근본적으로 뒤흔들고 있습니다. 그동안 당연하게 여겼던 '모든 데이터를 전역 스토어에 넣는' 관습에서 벗어나야 할 때입니다.

&nbsp;

---

&nbsp;

# 1. 서버 상태(Server State)의 서버 회귀

&nbsp;

과거에는 서버 데이터를 가져와 Redux나 Recoil 같은 스토어에 담아 관리했습니다. 하지만 RSC에서는 이 과정이 불필요해집니다.
- **데이터 페칭의 주체 변화**: 컴포넌트 자체가 서버에서 실행되므로 데이터를 직접 읽어옵니다.
- **캐싱의 주체 변화**: 클라이언트 사이드 캐싱 대신 Next.js의 내장 캐시나 서버 레벨의 데이터 관리가 그 역할을 대신합니다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 클라이언트 UI 상태로의 집중

&nbsp;

이제 클라이언트 사이드 상태 관리 라이브러리는 본연의 임무인 **'UI 상태'**에 집중하게 됩니다.
- **가벼운 상태 관리**: 데이터 동기화 기능이 빠진 **Zustand**나 **Jotai** 같은 작고 빠른 라이브러리가 선호됩니다.
- **순수 UI 로직**: 모달의 개폐, 폼 입력값, 현재 선택된 탭 정보 등 컴포넌트 간 공유가 필요한 순수 UI 정보만을 전역으로 관리합니다.

&nbsp;

&nbsp;

---

&nbsp;

# 3. 실전 엔지니어링 교훈: 번들 사이즈와 복잡도의 다이어트

&nbsp;

RSC를 도입한다는 것은 단순히 서버에서 렌더링한다는 것을 넘어, **클라이언트로 전송되는 자바스크립트 양을 획기적으로 줄인다**는 것을 의미합니다. 불필요한 전역 상태 관리 로직을 제거하고 서버 컴포넌트의 비중을 높임으로써, 더 빠르고 가벼운 프런트엔드 아키텍처를 설계할 수 있습니다.
