# [프런트엔드 08편] RSC와 상태 관리의 변화 — 서버 컴포넌트 도입 이후 클라이언트 전역 상태의 역할은 어떻게 재정의되는가?

&nbsp;

프런트엔드 개발의 역사는 '상태(State)를 어디에 둘 것인가'와의 투쟁이었다. Redux의 거대한 스토어부터 시작해, Recoil과 Zustand 같은 원자적 상태 관리까지 우리는 모든 데이터를 브라우저 메모리에 가두어 두려 노력했다. 하지만 **React Server Components(RSC)**의 등장은 이 십수 년의 관행을 뿌리째 흔들고 있다. 

&nbsp;

이제 데이터의 주권은 다시 서버로 넘어가고 있으며, 클라이언트 사이드 상태 관리는 그 어느 때보다 '가벼워질 것'을 요구받고 있다. 본 글에서는 RSC 아키텍처가 상태 관리 패러다임을 어떻게 바꾸고 있는지 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. 서버 상태의 서버 회귀: 데이터 페칭의 종말

&nbsp;

과거에는 서버에서 데이터를 가져오면, 이를 클라이언트 스토어에 복제(Sync)하고 캐싱하는 로직이 프런트엔드 코드의 절반 이상을 차지했다. 

&nbsp;

- **RSC의 데이터 직결**: 서버 컴포넌트는 DB나 외부 API에 직접 접근하여 데이터를 읽어온다. 클라이언트로 전송되는 것은 데이터가 아니라 이미 데이터가 채워진 '컴포넌트의 구조(추상화된 직렬화 데이터)'다. 
- **불필요해진 전역 캐시**: 브라우저 메모리에 수 MB의 JSON 데이터를 쌓아둘 필요가 사라졌다. 데이터 페칭과 캐싱의 책임은 이제 서버(Next.js Data Cache 등)가 전담하며, 클라이언트는 오직 그려진 결과물을 렌더링하기만 하면 된다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 클라이언트 UI 상태: 순수한 본연의 임무로

&nbsp;

서버 상태가 빠져나간 자리에 남은 것은 순수하게 브라우저 내에서만 유효한 **'UI 상태'**들이다.

&nbsp;

- **무거운 프레임워크의 퇴장**: 데이터 동기화 기능이 주력이었던 Redux나 Saga 같은 거대한 도구들은 RSC 환경에서 오버스펙이 된다. 
- **가벼운 도구의 부상**: 모달의 개폐, 폼 입력값의 임시 저장, 다크모드 설정 등 컴포넌트 간 공유가 필수적인 최소한의 상태만을 관리하기 위해 **Zustand**나 **Jotai** 같은 작고 빠른 라이브러리가 선호된다. 심지어 많은 경우, 리액트 내장 **Context API**만으로도 충분한 상황이 되었다.

# 3. 실전 엔지니어링 교훈: "상태 없는 프런트엔드"를 지향하라

&nbsp;

시니어 설계자라면 새로운 기술의 화려함 뒤에 숨겨진 **'트레이드오프'**를 보아야 한다.

1. **JS 번들 다이어트**: RSC를 쓰면서도 여전히 모든 데이터를 전역 스토어에 담고 있다면, 당신은 RSC의 가장 큰 이점인 '제로 클라이언트 JS'를 스스로 포기한 것이다. 상태 관리 라이브러리를 선택하기 전, "이 데이터가 반드시 클라이언트 메모리에 있어야 하는가?"를 먼저 자문하라.
2. **서버와 클라이언트의 명확한 경계**: 모든 것을 서버 컴포넌트로 만들 수는 없다. 상호작용이 잦은 영역은 과감히 클라이언트 컴포넌트로 분리하되, 서버에서 받은 데이터를 'Props'로 내려줄지 '상태'로 승격시킬지 신중히 결정해야 한다.

RSC 시대의 진정한 최적화는 **'브라우저가 기억해야 할 정보'를 최소한으로 줄이는 것**에서 시작된다. 가장 빠른 상태 관리는, 관리할 상태가 아예 없는 상태다.
