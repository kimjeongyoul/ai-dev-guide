# [프런트엔드 4편] 프런트엔드 배포 파이프라인 — 정적 자산(Static Assets)의 불변성과 트래픽 제어 전략

프런트엔드배포, 정적자산, CI/CD, BlueGreen, Canary, 불변성, 프런트엔드설계

&nbsp;

프런트엔드 애플리케이션은 HTML, JS, CSS와 같은 정적 자산(Static Assets)으로 구성되므로, 백엔드 서비스와는 다른 방식의 배포 전략이 필요합니다. 
 특히 대규모 사용자 환경에서 장애 위험을 최소화하기 위한 Blue-Green 및 Canary 배포를 정적 자산에 어떻게 적용할 수 있는지 분석합니다.

---

## 1. 정적 자산의 불변성과 버전 관리

프런트엔드 배포의 핵심은 파일의 불변성(Immutability)입니다. 빌드 시 생성되는 각 파일에 해시값을 부여하여 고유한 경로를 가지게 함으로써, 새로운 버전이 배포되어도 이전 버전의 자산이 즉시 삭제되지 않도록 관리해야 합니다.

*   Immutable Deploys: 새로운 버전 배포 시 기존 S3 버킷 등에 구 버전 파일을 일정 기간 유지합니다. 이는 현재 세션의 사용자가 페이지를 새로고침 하지 않은 상태에서 추가적인 Chunk 파일을 요청할 때 발생하는 404 에러를 방지합니다.

---

## 2. CDN 레벨에서의 트래픽 제어 전략

프런트엔드 자산은 대개 CDN(Content Delivery Network)을 통해 서빙되므로, 배포 전략 역시 CDN 설정과 밀접하게 연동됩니다.

### 2.1 Lambda@Edge 기반의 카나리(Canary) 배포
AWS CloudFront의 Lambda@Edge나 Cloudflare Workers를 활용하면 엣지 위치에서 실시간으로 요청을 가로채어 트래픽을 분산할 수 있습니다.
*   구현: 사용자 쿠키나 IP 주소, 또는 무작위 확률을 기반으로 특정 그룹의 요청을 신규 버전(Bucket B)의 `index.html`로 연결하고, 나머지는 기존 버전(Bucket A)으로 연결합니다.
*   이점: 서버 애플리케이션의 재배포 없이도 엣지 수준에서 실시간으로 신규 기능의 안정성을 검증할 수 있습니다.

### 2.2 Blue-Green 전환 로직
Blue-Green 배포의 경우, 신규 버전을 완전히 배포한 후 CDN의 원본 저장소(Origin) 경로를 한 번에 전환하거나, DNS 가중치 조절을 통해 점진적으로 이전합니다. 이 과정에서 정적 자산의 캐시 무효화(Invalidation) 전략이 적절히 수립되지 않으면 사용자가 구 버전의 HTML과 신규 버전의 JS가 혼용된 상태를 경험할 수 있으므로 주의가 필요합니다.

---

## 3. 안정적인 롤백 및 모니터링

*   Instant Rollback: `index.html` 파일은 용량이 작으므로 여러 버전을 관리하기 쉽습니다. 문제가 발생하면 CDN 설정에서 `index.html`이 가리키는 원본 경로를 이전 버전으로 즉시 되돌림으로써 롤백을 수행합니다.
*   런타임 에러 추적: Canary 배포 중인 특정 그룹에서만 발생하는 에러를 감지하기 위해 Sentry 등의 도구와 배포 버전을 연동하여 실시간 모니터링 대시보드를 구축해야 합니다.

결국 프런트엔드 배포 전략은 단순한 파일 전송을 넘어, CDN과 엣지 컴퓨팅을 활용한 정교한 트래픽 제어 아키텍처로 진화하고 있습니다.
