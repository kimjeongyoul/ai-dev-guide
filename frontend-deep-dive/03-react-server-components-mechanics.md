# React Server Components(RSC)의 동작 메커니즘과 페칭 전략의 변화

React Server Components(RSC)는 단순히 기존 SSR의 연장선이 아니라, 프런트엔드 애플리케이션의 런타임 구조를 서버와 클라이언트로 명확히 분리하는 새로운 패러다임입니다. 본 아티클에서는 RSC가 기존 SSR과 차별화되는 지점과 데이터 페칭 전략에 가져오는 변화를 심층 분석합니다.

---

## 1. RSC와 기존 SSR의 아키텍처적 차이

기존의 SSR은 서버에서 HTML을 생성하여 빠르게 전달하지만, 클라이언트에서 상호작용을 위해 전체 JavaScript 번들을 다운로드하고 실행(Hydration)해야 합니다.

*   번들 사이즈의 획기적 감소: RSC는 서버에서만 실행되는 컴포넌트입니다. 해당 컴포넌트 내부에서 사용하는 무거운 외부 라이브러리나 서버 전용 로직은 클라이언트 번들에 포함되지 않습니다. 이는 브라우저가 다운로드하고 파싱해야 할 코드량을 크게 줄여줍니다.
*   하이드레이션 제외: RSC의 결과물은 클라이언트로 전달될 때 HTML이 아닌 특수한 직렬화된 형식(RSC Payload)으로 전송되며, 클라이언트는 이를 해석하여 DOM을 업데이트합니다. 이 과정에서 서버 컴포넌트 부분은 하이드레이션 과정을 거치지 않으므로 메인 스레드 부하가 감소합니다.

---

## 2. 데이터 페칭 패러다임의 전환

RSC의 도입으로 클라이언트 중심의 데이터 페칭 전략이 서버 중심으로 이동하고 있습니다.

### 2.1 클라이언트 사이드 페칭의 한계
기존 방식에서는 `useEffect`나 React Query와 같은 훅을 사용하여 컴포넌트 렌더링 후 데이터를 당겨왔습니다. 이는 클라이언트-서버 간의 잦은 네트워크 왕복(Round-trip)과 상위 컴포넌트의 데이터 로딩이 끝나야 하위 컴포넌트가 로딩을 시작하는 Waterfall 현상을 유발합니다.

### 2.2 서버 내부 페칭과 Waterfall 해소
RSC 내부에서는 `async/await`를 사용하여 직접 데이터베이스나 외부 API에 접근할 수 있습니다. 데이터 페칭 로직이 데이터와 물리적으로 가까운 서버 단에서 실행되므로 지연 시간이 단축됩니다. 또한 여러 데이터 요청을 서버 내부에서 병렬로 처리하거나 하나의 요청으로 묶어 처리함으로써 클라이언트 단의 Waterfall 문제를 아키텍처적으로 해결합니다.

---

## 3. 실무 도입 시의 설계 고려사항

*   상태 관리와의 조율: RSC는 서버에서 실행되므로 브라우저의 전역 상태(Redux, Recoil 등)나 사용자 이벤트를 직접 다룰 수 없습니다. 따라서 상호작용이 필요한 부분은 Client Component로 분리하고, 데이터 중심의 구조는 Server Component로 설계하는 역할 분담이 중요합니다.
*   보안 전략: 서버 컴포넌트에서 직접 민감한 API를 호출할 때, 클라이언트로 노출되지 않아야 할 정보(API 키 등)가 환경 변수나 서버 전용 모듈(`server-only`)을 통해 엄격히 관리되어야 합니다.
*   캐싱과 재검증(Revalidation): RSC 환경에서는 Next.js 등 프레임워크가 제공하는 데이터 캐싱 메커니즘을 적극 활용하여 불필요한 서버 연산을 줄이고 최신 데이터를 유지하는 전략을 수립해야 합니다.
