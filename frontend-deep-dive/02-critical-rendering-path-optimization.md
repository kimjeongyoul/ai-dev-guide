# [프런트엔드 2편] Critical Rendering Path 최적화 — 브라우저 렌더링 파이프라인 이해를 통한 LCP 개선 전략

CRP최적화, 브라우저렌더링, 웹성능, LCP개선, DOM파싱, 성능최적화, 프런트엔드설계

&nbsp;

웹 성능 지표 중 하나인 LCP(Largest Contentful Paint)는 사용자 경험을 결정짓는 핵심적인 요소입니다. 
 단순히 이미지 크기를 줄이는 것을 넘어, 브라우저의 Critical Rendering Path(CRP)를 이해하고 각 단계의 병목을 제거하는 고도화된 최적화 전략이 필요합니다.

---

## 1. 리소스 우선순위 지정과 렌더링 차단 해제

브라우저가 HTML을 파싱하는 과정에서 JS나 CSS와 같은 외부 리소스를 만나면 파싱이 중단됩니다. 이를 관리하는 것이 CRP 최적화의 첫걸음입니다.

### 1.1 fetchpriority를 통한 리소스 로딩 제어
LCP 대상이 되는 핵심 이미지나 폰트 리소스에 `fetchpriority="high"`를 부여하여 브라우저의 다운로드 우선순위를 높여야 합니다. 반면 초기 렌더링에 중요하지 않은 이미지에는 `loading="lazy"`를 적용하여 불필요한 네트워크 점유를 피합니다.

### 1.2 비차단 리소스 로딩 전략
초기 뷰포트 렌더링에 필수적이지 않은 JavaScript는 `defer`나 `async` 속성을 활용하여 실행 시점을 늦춰야 합니다. CSS의 경우 Critical CSS 기법을 사용하여 초기 렌더링에 필요한 스타일만 인라인화하고, 나머지 스타일은 비동기적으로 로딩함으로써 렌더링 차단 시간을 최소화합니다.

---

## 2. 서버 응답 시간과 데이터 전달 최적화

초기 HTML 데이터가 도달하는 시간(TTFB)은 모든 렌더링 지표의 기본이 됩니다.

### 2.1 서버 사이드 렌더링(SSR) 및 스트리밍 활용
SSR을 통해 완성된 HTML을 전달하면 브라우저가 JS 번들을 실행하기 전에도 콘텐츠를 표시할 수 있습니다. 최근에는 HTML을 조각 단위로 순차 전달하는 Streaming SSR을 도입하여, 전체 페이지가 준비되기 전에도 상단 콘텐츠부터 빠르게 렌더링하는 전략이 유효합니다.

### 2.2 폰트 최적화와 레이아웃 시프트 방지
폰트 로딩 시 발생하는 텍스트 미표시(FOIT)나 대체 폰트 표시(FOUT) 현상은 사용자 경험을 저해합니다. `font-display: swap` 속성과 함께 폰트 파일을 미리 연결하는 `preload` 전략을 사용하고, 웹 폰트 로딩 후의 레이아웃 시프트(CLS)를 방지하기 위해 폰트 메트릭을 조정하는 최적화가 수반되어야 합니다.

---

## 3. 현대적 렌더링 아키텍처 도입

하이드레이션(Hydration)은 클라이언트 사이드에서 메인 스레드를 장기간 점유하여 상호작용 지연을 유발할 수 있습니다.

*   Island Architecture: 페이지 전체를 하이드레이션 하는 대신, 상호작용이 필요한 특정 부분(Islands)만 독립적으로 활성화하여 메인 스레드 부하를 줄입니다.
*   Selective Hydration: 리액트 18 등에서 지원하는 기능으로, 사용자가 상호작용을 시도하는 부분부터 우선적으로 하이드레이션 하여 실질적인 체감 성능을 개선합니다.

결국 LCP 개선은 브라우저가 네트워크 환경과 하드웨어 리소스를 얼마나 효율적으로 사용하여 가장 중요한 콘텐츠를 먼저 그려내느냐에 달려 있습니다.
