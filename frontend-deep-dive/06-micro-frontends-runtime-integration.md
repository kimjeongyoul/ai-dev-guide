# [프런트엔드 06편] Micro-Frontends 런타임 통합 — 팀 간 배포 독립성을 확보하기 위한 Module Federation 설계 전략은?

&nbsp;

"하나의 거대한 모놀리스가 팀의 속도를 늦추고 있다." 프런트엔드 애플리케이션이 비대해지면 빌드 시간은 늘어나고, 작은 수정 하나에도 전체 서비스를 재배포해야 하는 리스크를 안게 된다. 여러 팀이 하나의 프로젝트에 매달릴수록 코드 충돌은 일상이 되고, 배포 기차는 점점 무거워진다. 이 병목을 뚫기 위해 등장한 것이 바로 **Micro-Frontends(MFE)**다. 

&nbsp;

본 글에서는 MFE 구현의 정점으로 불리는 Webpack 5의 **Module Federation**을 중심으로, 어떻게 런타임에 여러 앱을 결합하면서도 독립적인 배포 자율성을 유지할 수 있는지 그 아키텍처적 비결을 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. Module Federation: 빌드 타임 의존성의 종말

&nbsp;

과거의 MFE는 NPM 패키지 방식(빌드 타임 통합)이나 iframe을 사용했다. 전자는 독립 배포가 불가능했고, 후자는 성능과 사용자 경험이 최악이었다. Module Federation은 이 둘의 장점만을 취해 **'런타임 공유 의존성'**이라는 개념을 도입했다.

&nbsp;

- **Host와 Remote의 역동적 관계**: 메인 앱(Host)은 필요한 시점에만 마이크로 앱(Remote)의 코드를 브라우저로 가져온다. Remote 앱의 코드가 수정되어 배포되면, Host 앱은 재빌드 없이도 즉시 최신 기능을 반영한다.
- **Shared Dependencies**: 모든 마이크로 앱이 각자 React를 로드한다면 번들 크기는 폭발할 것이다. Module Federation은 브라우저에 이미 로드된 라이브러리가 있다면 이를 재사용하고, 버전이 맞지 않을 때만 새로 로드하는 영리한 의존성 관리 메커니즘을 제공한다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 실전의 벽: 버전 지옥과 장애 전파

&nbsp;

런타임 통합은 강력하지만, 그만큼 위험하다. 

&nbsp;

- **버전 불일치 (The Singleton Problem)**: React와 같이 싱글톤으로 동작해야 하는 라이브러리의 버전이 팀마다 다를 경우, 런타임에 예상치 못한 에러가 발생한다. 이를 위해 엄격한 **Semantic Versioning(SemVer)** 규약을 세우고, `shared` 설정을 통해 허용 범위를 명확히 정의해야 한다.
- **장애 격리 (Error Boundaries)**: 특정 팀이 배포한 코드에 버그가 있다면 전체 화면이 화이트아웃될 수 있다. 각 마이크로 앱의 진입점에 강력한 **Error Boundary**를 배치하여, 특정 기능이 죽더라도 사용자에게는 "잠시 후 이용해 주세요"라는 메시지와 함께 나머지 기능은 정상 작동하도록 설계해야 한다.

# 3. 실전 엔지니어링 교훈: 기술보다 조직의 결합도가 우선이다

&nbsp;

MFE는 단순한 기술 도입이 아니라 **'조직 구조의 반영'**이다.

1. **도메인 경계 설정**: 기술적인 기준으로 앱을 나누지 마라. 비즈니스 도메인(예: 주문, 결제, 상품)을 기준으로 팀을 나누고 그 경계에 맞춰 마이크로 앱을 설계해야 진정한 배포 자율성이 생긴다.
2. **공통 자산의 최소화**: 공통 컴포넌트 라이브러리가 너무 비대해지면, 결국 그 라이브러리가 새로운 모놀리스가 되어 모든 팀의 발목을 잡는다. '공유'보다 '복제'가 나을 때가 있음을 시니어 설계자는 직관적으로 알아야 한다.

결국 MFE 아키텍처의 성공은 **"얼마나 잘 합치느냐"가 아니라 "얼마나 깔끔하게 각자의 길을 가느냐"**에 달려 있다.
