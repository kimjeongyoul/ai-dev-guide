# [프런트엔드 06편] Micro-Frontends 런타임 통합 — 팀 간 배포 독립성을 확보하기 위한 Module Federation 설계 전략은?

ModuleFederation, 런타임통합, Webpack5, 의존성관리, 장애격리, MFE, 프런트엔드설계

&nbsp;

대규모 조직에서 프런트엔드 앱이 비대해지면 팀 간 개발 간섭과 배포 지연이 발생합니다. 이를 해결하기 위해 각 기능을 독립적으로 배포하고 브라우저에서 하나로 합치는 **Micro-Frontends** 런타임 통합 전략을 분석합니다.

&nbsp;

---

&nbsp;

# 1. Module Federation: 빌드 없는 통합

&nbsp;

Webpack 5의 Module Federation은 MFE 구현의 게임 체인저입니다.
- **런타임 로딩**: 각 팀은 자신의 마이크로 앱을 독립적인 인프라에 배포하고, 메인 앱은 이를 실행 시점에 동적으로 가져옵니다.
- **배포 독립성**: 특정 기능의 버그 수정이나 업데이트를 위해 메인 앱 전체를 다시 빌드하고 배포할 필요가 없습니다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 공유 의존성(Shared Dependencies)과 버전 관리

&nbsp;

런타임 통합의 가장 큰 적은 라이브러리 중복 로드와 버전 충돌입니다.
- **싱글톤 설정**: React나 디자인 시스템 라이브러리를 Shared로 설정하여 브라우저에 단 한 번만 로드되도록 강제합니다.
- **엄격한 SemVer**: 마이크로 앱 간의 인터페이스 변화를 Semantic Versioning 정책으로 관리하여, 업데이트가 다른 팀의 기능을 깨뜨리지 않도록 보장합니다.

&nbsp;

&nbsp;

---

&nbsp;

# 3. 실전 엔지니어링 교훈: 에러 격리와 사용자 경험

&nbsp;

독립적인 배포는 독립적인 장애를 의미합니다.
- **에러 바운더리**: 특정 마이크로 앱에서 런타임 에러가 발생해도 전체 서비스가 화이트아웃되지 않도록, 각 진입점에 견고한 에러 바운더리를 배치합니다.
- **일관된 UX**: 기술적으로는 분리되어 있어도 사용자에게는 하나의 서비스처럼 느껴지도록 디자인 시스템과 공통 에셋 관리 체계를 구축해야 합니다.
