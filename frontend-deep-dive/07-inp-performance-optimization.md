# [프런트엔드 07편] INP(Interaction to Next Paint) 최적화 — 사용자의 모든 입력에 즉각 반응하는 매끄러운 경험을 어떻게 만들 것인가?

&nbsp;

"분명 클릭했는데 왜 반응이 없지?" 사용자가 웹사이트에서 느끼는 가장 큰 좌절감 중 하나는 상호작용의 지연이다. Google은 이러한 사용자 경험을 정량화하기 위해 기존의 FID(First Input Delay)를 폐기하고, 2024년부터 새로운 핵심 웹 지표로 **INP**를 도입했다. 첫 입력만 측정하던 과거와 달리, INP는 사용자가 페이지를 떠날 때까지 발생하는 **모든 상호작용**을 감시한다. 

&nbsp;

본 글에서는 브라우저의 렌더링 파이프라인을 점유하는 '긴 작업(Long Task)'의 정체를 파악하고, 사용자 입력에 즉각 응답하기 위한 하이레벨 최적화 전략을 다룬다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. INP의 핵심: 메인 스레드를 해방하라

&nbsp;

브라우저의 메인 스레드는 싱글 스레드다. 자바스크립트 실행, 스타일 계산, 레이아웃, 페인팅을 모두 혼자 처리한다. 만약 자바스크립트 로직이 500ms 동안 CPU를 붙잡고 있다면, 그동안 사용자가 누른 버튼은 아무런 반응도 얻지 못한다. 이것이 INP 저하의 근본 원인이다.

&nbsp;

- **Yielding 전략**: 큰 작업을 작은 덩어리로 쪼개야 한다. `setTimeout`이나 `requestIdleCallback`을 사용할 수도 있지만, 최신 브라우저에서는 **`scheduler.yield()`**를 통해 현재 실행 중인 작업을 멈추고 브라우저에게 "급한 사용자 입력이 있다면 먼저 처리해"라고 양보할 수 있다.
- **Web Worker의 활용**: 복잡한 데이터 가공이나 대규모 정렬 로직은 메인 스레드에 둘 이유가 없다. 이를 Web Worker로 이전함으로써 메인 스레드는 오직 UI 업데이트와 사용자 응답에만 집중하게 만든다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 리액트 렌더링과 우선순위 지능

&nbsp;

현대의 프레임워크, 특히 리액트는 INP 개선을 위한 강력한 무기를 제공한다.

&nbsp;

- **Concurrent Rendering**: 리액트 18부터 도입된 동시성 모드는 렌더링 도중 사용자 입력이 들어오면 현재 진행 중인 렌더링을 일시 중단하고 입력부터 처리한다.
- **useTransition과 useDeferredValue**: 모든 상태 업데이트가 평등할 필요는 없다. 입력 필드의 글자는 즉시 보여줘야 하지만(Urgent), 아래의 검색 결과 목록 업데이트는 조금 늦어도 된다(Transition). 이러한 **우선순위 제어**는 메인 스레드의 부하를 분산시켜 체감 반응성을 극적으로 높인다.

&nbsp;

&nbsp;

---

&nbsp;

# 3. 실전 엔지니어링 교훈: 200ms의 법칙

&nbsp;

인간이 지연을 느끼지 못하는 한계치는 약 100ms이며, 200ms를 넘어서는 순간 '느리다'고 인지하기 시작한다.

1. **상태 업데이트의 최소화**: 불필요한 리렌더링을 일으키는 거대한 Context API나 전역 상태 관리자를 경계하라. 원자적(Atomic) 상태 관리를 통해 변경 범위를 국소화하는 것이 INP 최적화의 첫걸음이다.
2. **비침습적 모니터링**: 개발 환경에서는 잘 작동하던 페이지가 저사양 기기나 네트워크 환경에서는 INP 지옥이 될 수 있다. 실제 사용자 환경(RUM 데이터)에서 발생하는 INP 지표를 지속적으로 추적하고, 병목이 발생하는 특정 컴포넌트를 격리하여 수정해야 한다.

반응성은 프런트엔드 품질의 자존심이다. 사용자의 클릭에 0.1초 만에 화답하는 웹사이트는 기술력을 넘어 **사용자에 대한 존중**의 표현이다.
