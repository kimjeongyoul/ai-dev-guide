# [프런트엔드 07편] INP(Interaction to Next Paint) 최적화 — 사용자의 모든 입력에 즉각 반응하는 매끄러운 경험을 어떻게 만들 것인가?

INP최적화, 웹성능, 반응성, 메인스레드, Yielding, WebWorker, 프런트엔드설계

&nbsp;

Google이 새롭게 정의한 핵심 웹 지표인 **INP**는 사용자의 입력이 시각적으로 반영되기까지의 전체 지연 시간을 측정합니다. 단순히 첫 입력만 보던 과거의 지표를 넘어, 페이지 전체의 반응성을 극대화하기 위한 최적화 전략을 다룹니다.

&nbsp;

---

&nbsp;

# 1. 메인 스레드 점유 해소: Yielding 전략

&nbsp;

INP를 저하시키는 가장 큰 요인은 브라우저의 메인 스레드를 장시간 독점하는 '긴 작업(Long Task)'입니다.

## 2-1. 작업 쪼개기 (scheduler.yield)
복잡한 연산이나 대용량 데이터 렌더링을 수행할 때, 브라우저가 사용자 입력을 처리할 수 있도록 제어권을 반환해야 합니다. `setTimeout`이나 최신 `scheduler.yield()`를 활용해 작업을 작은 단위로 쪼개어 실행합니다.

## 2-2. Web Worker로 무거운 연산 이전
UI 렌더링과 관련 없는 순수 데이터 가공 로직은 **Web Worker**로 분리합니다. 이를 통해 메인 스레드는 오직 사용자와의 상호작용과 페인팅에만 집중할 수 있게 됩니다.

# 2. 상태 업데이트와 렌더링 우선순위

&nbsp;

- **useTransition 활용**: React 환경에서 긴급하지 않은 상태 업데이트를 전환(Transition)으로 표시합니다. 이를 통해 사용자 입력이 발생했을 때 렌더링 작업을 즉시 중단하고 입력에 먼저 반응하도록 우선순위를 조정합니다.
- **리렌더링 최소화**: 불필요한 DOM 조작을 방지하고, 빈번한 이벤트에 대해서는 적절한 디바운싱/스로틀링을 적용하여 메인 스레드 부하를 줄입니다.

# 3. 실전 엔지니어링 교훈: 반응성이 곧 신뢰다

&nbsp;

사용자는 버튼을 눌렀을 때 200ms 이상의 지연이 발생하면 시스템이 멈췄다고 느낍니다. INP 최적화는 단순히 수치를 개선하는 작업이 아니라, 사용자와 프로덕트 사이의 **'실시간성'**과 **'신뢰'**를 구축하는 엔지니어링의 핵심입니다.
