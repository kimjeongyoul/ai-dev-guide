# [실전 AIOps 5편] 포스트모템(Post-mortem) 자동화 — 슬랙 대화와 메트릭을 취합하는 장애 보고서 에이전트

&nbsp;

"자, 이제 템플릿에 맞춰서 장애 보고서(Post-mortem) 써주세요."

&nbsp;

서비스 장애를 복구하느라 밤을 새운 SRE(Site Reliability Engineering) 엔지니어에게 가장 잔인한 순간은 장애 그 자체가 아니라, 다음 날 아침 출근해서 벌어지는 '문서 작업'이다. 장애가 언제 시작되었고 누가 처음 인지했으며, 슬랙(Slack) 채널에서 무슨 대화가 오갔고, 최종적으로 어떤 명령어를 쳐서 복구했는지. 이 모든 타임라인을 수작업으로 복기하여 Confluence나 Notion에 정리하는 작업은 평균 2~3시간이 소요되는 고통스러운 노가다다.

&nbsp;

하지만 장애 대응 과정의 모든 기록은 이미 슬랙 채널, Datadog 대시보드, 그리고 K8s Audit Log에 파편화되어 남아 있다. 본 글에서는 이 흩어진 메타데이터와 엔지니어들의 텍스트 대화 내역을 LLM 에이전트가 긁어모아, 단 1분 만에 완벽한 마크다운(Markdown) 포맷의 **비난 없는 포스트모템(Blameless Post-mortem)** 문서를 자동 생성해 내는 AIOps 문서화 파이프라인의 아키텍처를 해부한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. 아키텍처: 파편화된 컨텍스트의 수집 (Data Ingestion)

&nbsp;

훌륭한 포스트모템의 핵심은 '정확한 타임라인(Timeline)'이다. 에이전트는 장애 알람이 해제(Resolved)된 순간을 트리거(Trigger)로 삼아, 지난 N시간 동안의 모든 인프라 컨텍스트를 수집하는 백그라운드 워커를 실행한다.

&nbsp;

## 1-1. Slack API를 통한 대화 내역 추출
장애 전용 채널(예: `#incident-payment-api`)에서 발생한 모든 메시지와 스레드(Thread) 댓글을 시간순으로 긁어온다.
단순히 텍스트만 가져오는 것이 아니라, 엔지니어가 공유한 로그 캡처 이미지, Datadog 차트 링크, 스크립트 실행 결과 코드 블록 등을 분류하여 메타데이터화한다.

&nbsp;

## 1-2. 시스템 메트릭과 Audit Log 스냅샷
슬랙 대화만으로는 기술적 깊이가 부족하다. 
- **APM 데이터**: Datadog API를 찔러 장애 발생 전후 1시간의 CPU, Memory, HTTP 5xx 에러율의 임계치 돌파 시점과 최고점(Peak) 데이터를 수치(JSON)로 추출한다.
- **K8s & Git 이력**: 해당 시간대에 어떤 파드(Pod)가 재시작되었는지(K8s Event Log), 그리고 장애 발생 직전에 어떤 PR이 머지(Merge)되었는지(GitHub API)를 긁어온다.

&nbsp;

이 세 가지 소스(대화, 지표, 변경 이력)가 모이면, 장애를 둘러싼 '현장 검증 데이터'가 100% 확보된 것이다.

&nbsp;

# 2. LLM 파이프라인: 무질서한 텍스트의 구조화 (Sense-Making)

&nbsp;

수집된 데이터는 그 자체로 거대한 노이즈(Noise)다. 슬랙 대화에는 "아 이거 또 이러네", "제가 볼게요", "커피 좀 마시고 올게요" 같은 무의미한 감정 텍스트가 절반을 차지한다. 
LLM의 가장 강력한 무기인 **요약(Summarization)과 개체명 인식(NER)** 능력을 활용하여 이 텍스트 덩어리를 정형화된 보고서 템플릿으로 변환해야 한다.

&nbsp;

## 2-1. System Prompt 설계: SRE 리포터 페르소나
에이전트에게는 구글 SRE 워크북 수준의 엄격한 포스트모템 작성 가이드라인을 프롬프트로 주입한다.

&nbsp;

```python
prompt = f"""
당신은 최고 수준의 SRE 테크니컬 라이터입니다. 
제공된 슬랙 대화 타임라인, APM 지표 스냅샷, 인프라 변경 이력을 분석하여 [Blameless Post-mortem] 마크다운 문서를 작성하십시오.

[분석 대상 데이터]
1. Slack 대화 기록: {slack_threads_json}
2. 시스템 지표(Datadog): {metrics_snapshot_json}
3. K8s/Git 변경 이력: {events_history_json}

[작성 규칙]
1. Timeline: 장애 인지 시점부터 복구 시점까지 분(Minute) 단위로 엔지니어의 핵심 액션(명령어 실행 등)만 추려내어 작성하십시오. 무의미한 인사말은 제외합니다.
2. Root Cause: 엔지니어들의 대화 내용과 시스템 지표를 대조하여, 가장 근본적인 원인 1가지를 기술적으로 서술하십시오.
3. Action Items: 슬랙 대화 중 "앞으로는 ~게 해야겠네요", "~에 알람을 추가합시다"라고 언급된 재발 방지 대책을 추출하여 체크리스트(TODO) 형태로 정리하십시오.
4. 어조: 특정 개인을 비난하는 표현(예: "김개발이 코드를 잘못 짜서")을 철저히 배제하고, 시스템의 관점(예: "리뷰 파이프라인 누락으로 인해")으로 순화하여 작성하십시오.
"""
```

&nbsp;

이 프롬프트를 통과하는 순간, "A님이 DB 덤프 떴고 B님이 롤백했어요"라는 거친 대화가 "03:45 KST - DB 커넥션 풀 고갈 확인 및 힙 덤프 생성 완료. 03:50 KST - 이전 안정화 버전(v1.2.3)으로 롤백 배포 완료"라는 정제된 엔지니어링 타임라인으로 180도 탈바꿈한다.

# 3. 실전 적용: 지식의 자산화 (Knowledge Base)

&nbsp;

생성된 마크다운 리포트는 즉시 슬랙 채널에 초안(Draft) 형태로 공유된다. 
리드 엔지니어는 초안을 가볍게 훑어보고 "Root Cause 설명에 인덱스 타임아웃 내용 덧붙여줘"라고 봇에게 코멘트만 달면, 에이전트는 즉각 내용을 보완한 뒤 사내 Confluence API나 Notion API를 찔러 **공식 장애 보고서 페이지를 자동으로 발행**한다.

&nbsp;

이 아키텍처의 진짜 위력은 단순히 문서 작성 시간을 2시간에서 2분으로 줄인 것에 그치지 않는다. 
이렇게 강제적으로, 그리고 일관된 포맷으로 생성된 수십 개의 포스트모템 문서들은 다시 Vector DB로 들어가 **사내 SRE 챗봇의 가장 훌륭한 RAG(Retrieval-Augmented Generation) 지식 베이스(Knowledge Base)**가 된다. 
6개월 뒤 비슷한 장애가 터졌을 때, 엔지니어가 "이거 예전에도 이랬던 것 같은데?"라고 봇에게 묻는 순간, 봇은 "6개월 전 장애 보고서 #104에 따르면, 당시에도 Redis 캐시 만료 동기화 버그가 원인이었으며 조치 방법은 다음과 같습니다"라며 과거의 삽질을 1초 만에 소환해 내는 완벽한 선순환 고리가 완성되는 것이다.

# 결론: 엔지니어는 시스템을 고치고, AI는 역사를 기록한다

&nbsp;

장애는 피할 수 없다. 중요한 것은 그 장애로부터 우리 조직이 얼마나 뼈아픈 교훈을 얻어내고 시스템을 단단하게 만드느냐에 있다. 하지만 인간은 망각의 동물이자 귀찮음을 피하려는 존재이기에, 문서화는 늘 뒷전으로 밀린다.

&nbsp;

데이터 수집부터 요약, 타임라인 구성, 그리고 재발 방지 대책의 구조화까지. 포스트모템 자동화 에이전트는 조직의 가장 훌륭한 서기(Scribe)이자, 똑같은 실수를 반복하지 않게 만들어주는 시스템의 뇌(Brain)다. AIOps의 완성은 장애를 빠르게 고치는 것을 넘어, 그 장애를 조직의 영구적인 기술 자산으로 번역해 내는 데 있음을 명심하라.

&nbsp;

이것으로 백엔드 시스템의 모니터링부터 로그 분석, 자동 복구, 카오스 엔지니어링, 그리고 포스트모템 자동화에 이르는 **[실전 AIOps 시리즈 1~5편]**의 대장정을 마칩니다. AI가 단순한 코딩 보조 도구를 넘어, 거대한 분산 시스템의 신뢰성을 지키는 수석 SRE 엔지니어로 진화하는 아키텍처 설계에 이 기록들이 영감이 되길 바랍니다.
