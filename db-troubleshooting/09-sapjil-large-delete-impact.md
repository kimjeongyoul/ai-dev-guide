# [실전 삽질 5편] 대량 데이터 삭제(Delete)의 역습 — Undo Log와 MVCC 병목의 파국

&nbsp;

"스토리지 용량이 90%를 넘었습니다. 시스템 알람 지표가 임계치입니다. 빨리 불필요한 로그 데이터 좀 비워주세요!"

&nbsp;

인프라 모니터링 파트에서 긴급 호출이 떨어졌다. 문제가 된 테이블은 접속 이력이나 결제 상태 변경 등을 추적하기 위해 쌓아둔 `system_audit_logs` 테이블이었다. 로우(Row) 수만 8천만 건을 돌파하며 거대해져 있었고, 우리는 "오래된 데이터는 의미가 없으니 2년 전 데이터 5천만 건만 쿨하게 날려버리자"는 결정을 내렸다.

&nbsp;

```sql
-- 평범하고 무식한 삭제 쿼리의 실행
DELETE FROM system_audit_logs WHERE created_at < '2022-01-01 00:00:00';
```

&nbsp;

엔터를 누른 순간 용량이 쾌적하게 비워질 것이라는 순진한 기대는 불과 1분 만에 산산조각 났다. 삭제 쿼리가 실행되는 동안 스토리지는 줄어들기는커녕 오히려 더 빠른 속도로 폭증하기 시작했고, 디스크 I/O가 한계에 다다르며 해당 DB를 함께 쓰던 모든 메인 서비스의 트랜잭션이 멈춰 섰다(Deadlock 및 Timeout). 데이터를 비우기 위해 던진 쿼리가 거꾸로 시스템의 숨통을 조여버린 이 역설적인 참사는, InnoDB 스토리지 엔진의 핵심 아키텍처인 **Undo Log**와 **MVCC** 메커니즘을 제대로 이해하지 못해 벌어진 대형 참사였다.

&nbsp;

&nbsp;

&nbsp;

# 1. 삭제의 이면: 데이터는 한 번에 지워지지 않는다

&nbsp;

트랜잭션을 완벽하게 지원하는 현대 RDBMS(MySQL InnoDB 등) 환경에서 `DELETE`는 물리적 디스크의 데이터 블록을 그 즉시 파괴하는 작업이 아니다. 

&nbsp;

## 1-1. Rollback을 대비한 그림자 공간: Undo Log
가장 치명적인 오해는 "지우면 용량이 줄어든다"고 믿는 것이다.
5천만 건의 데이터를 지우는 도중에 서버 전원이 나가거나 명시적인 `ROLLBACK` 명령이 떨어진다면? 데이터베이스는 반드시 삭제 전의 원상태로 데이터를 완벽하게 복구해내야 한다. 이를 위해 InnoDB 엔진은 데이터에 삭제 마킹(Delete Mark)을 하기 직전, **원본 데이터의 완벽한 복제본을 별도의 시스템 테이블스페이스(또는 Undo 테이블스페이스)인 Undo Log 영역에 고스란히 복사**해 둔다.

&nbsp;

5천만 건의 데이터를 한 번의 거대한 트랜잭션 안에서 지운다는 것은, 일시적으로 디스크 공간 내에 원본 데이터와 복제본(Undo) 데이터를 이중으로 떠안게 됨을 의미한다. 만약 대상 테이블에 여러 개의 무거운 인덱스(Index)까지 걸려 있다면, 각 인덱스 엔트리의 변경 전 상태 정보까지 모두 기록해야 하므로 스토리지 폭증의 가속도는 기하급수적으로 치솟는다. 데이터를 지우기 위해 던진 쿼리가 역설적으로 디스크 풀(Disk Full) 장애를 직접 유발하는 것이다.

&nbsp;

&nbsp;

&nbsp;

# 2. 성능 파괴의 근원: MVCC와 Purge Lag 현상

&nbsp;

용량 폭증을 넘어 서버 전체의 트랜잭션을 마비시키는 진짜 원인은 동시성 제어 메커니즘인 MVCC(Multi-Version Concurrency Control)에 있다.

&nbsp;

## 2-1. 과거를 붙잡고 있는 다른 트랜잭션들
데이터베이스는 동시에 여러 사용자의 쿼리를 처리한다. 내가 5천만 건의 과거 데이터를 삭제하고 있는 그 순간에도, 다른 어떤 커넥션에서는 긴 시간을 들여 테이블 전체의 통계를 내는(SELECT) 무거운 리포트 쿼리를 실행 중일 수 있다.
데이터 일관성 원칙(Repeatable Read)에 따라, 조회를 수행 중인 트랜잭션은 자신이 쿼리를 시작한 시점의 '정상적이고 깨끗한(삭제 전)' 데이터를 볼 권리가 있다. 이를 보장하기 위해 데이터베이스 엔진은 백그라운드 가비지 컬렉터(Purge Thread)가 삭제 처리된 낡은 로우들을 디스크에서 실제로 영구 삭제하는 작업을 **'강제로 중단'**시킨다.

&nbsp;

## 2-2. History List Length 폭발과 시스템 다운
결과적으로, 삭제 쿼리는 끝없이 밀려드는 데 찌꺼기 데이터는 정리되지 못하고 Undo Log에 계속 축적되는 현상(Purge Lag)이 발생한다. 
모니터링 지표 중 트랜잭션의 노화 상태를 보여주는 `History list length` 수치가 수백만 단위로 폭증하게 되고, 옵티마이저와 스토리지 엔진은 데이터를 조회할 때마다 수많은 데이터의 '버전(Version)' 체인을 거슬러 올라가며 유효성을 검사해야 하므로 CPU 코어가 100%에 고정되며 서비스가 완전 마비 상태에 이르게 된다.

# 3. 안전한 대규모 마이그레이션 아키텍처 (Chunking & Swapping)

&nbsp;

디스크와 CPU를 죽이지 않고 수천만 건의 데이터를 우아하게 썰어내는 실무적인 엔지니어링 전략은 크게 두 가지로 나뉜다.

&nbsp;

## 3-1. 전략 A: 청크 단위의 호흡 조절 (Chunking Delete)
단일 거대 트랜잭션을 잘게 쪼개어 시스템이 숨통을 틔울 틈(Purge)을 부여하는 가장 고전적이면서도 안전한 방식이다.

&nbsp;

```sql
-- 한 번에 5천 건씩만 지우고 커밋을 반복하는 로직을 애플리케이션이나 프로시저에 구현
WHILE (데이터가 남아있을 때까지) DO
    DELETE FROM system_audit_logs 
    WHERE created_at < '2022-01-01 00:00:00' 
    ORDER BY id ASC  -- PK 순차 접근으로 데드락 방지
    LIMIT 5000;      -- 트랜잭션 덩치를 제한하여 Undo Log 폭증 방어
    
    -- 강제로 1초 휴식 (Sleep)
    -- 이 대기 시간 동안 InnoDB의 백그라운드 스레드가 눈치껏 Undo Log를 비우고 버퍼 풀을 정리함
    DO SLEEP(1);
END WHILE;
```

&nbsp;

이 방식은 수 일이 걸릴 정도로 오래 걸리지만, 서비스의 메인 비즈니스 트랜잭션에 거의 부하를 주지 않아 무중단 라이브 환경에서 가장 선호된다.

&nbsp;

## 3-2. 전략 B: 남길 데이터 옮기기와 포인터 스왑 (Create-Insert-Swap)
만약 전체 테이블의 80~90%를 삭제해야 하는 극단적인 상황이라면, `DELETE` 쿼리를 쓰는 것 자체가 바보 같은 짓이다. 삭제 후 발생하는 파편화(Fragmentation)로 인해 디스크 빈 공간이 OS로 반환되지 않아 별도의 `OPTIMIZE TABLE` 연산까지 돌려야 하는 이중고에 빠지기 때문이다.
이때는 아예 새 집을 지어서 '살릴 데이터'만 옮긴 뒤 문패를 바꿔 다는 스왑 전략이 백배 빠르고 깔끔하다.

&nbsp;

```sql
-- 1. 동일한 구조의 빈 테이블 생성 (인덱스 구조 포함)
CREATE TABLE logs_new LIKE system_audit_logs;

-- 2. 남겨둘 최신 데이터(2022년 이후)만 새 테이블에 빠르게 복사 (INSERT 속도가 DELETE보다 훨씬 빠르고 안정적임)
INSERT INTO logs_new SELECT * FROM system_audit_logs WHERE created_at >= '2022-01-01 00:00:00';

-- 3. 테이블 이름 순식간에 교체 (원자적 메타데이터 변경)
RENAME TABLE system_audit_logs TO logs_old, logs_new TO system_audit_logs;

-- 4. 한적한 새벽에 버려진 옛날 테이블을 영구 삭제 (순식간에 디스크 반환됨)
DROP TABLE logs_old;
```

&nbsp;

이 스왑 패턴은 며칠이 걸릴 삭제 작업을 단 몇십 분 만에 완벽한 무중단 상태로 종결시키는 최고의 아키텍처다.

# 결론: 데이터의 '삶'만큼 '죽음'도 철저히 설계하라

&nbsp;

데이터를 삽입(`INSERT`)하고 조회(`SELECT`)하는 쿼리의 성능은 누구나 고민하지만, 데이터를 파기(`DELETE`)하는 쿼리의 부하를 고민하는 엔지니어는 드물다. 하지만 트랜잭션 시스템에서 데이터의 파기는 그 어떤 연산보다 디스크 I/O와 메모리 리소스를 무자비하게 갉아먹는 파괴적인 작업이다.

&nbsp;

대량의 데이터를 삭제하기 전에는 반드시 Undo 테이블스페이스의 여유 공간을 점검하고, 운영 중인 롱 트랜잭션(Long Transaction)과의 충돌 여부를 가늠하며, 작업 단위를 잘게 쪼개어 시스템 엔진의 숨통을 배려하는 호흡 조절 로직을 설계해야 한다. 쿼리 한 줄로 해결될 것 같아 보이는 단순함 속에 숨어 있는 데이터베이스 엔진의 복잡한 메커니즘을 꿰뚫어 보는 통찰력, 그것이 대규모 분산 시스템을 지탱하는 시니어 백엔드 엔지니어의 핵심 역량이다.
