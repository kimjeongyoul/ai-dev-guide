# [실전 삽질 9편] 레거시 청산의 공포 — AI 기반 섀도우 트래픽(Shadow Traffic) 검증과 무중단 마이그레이션

&nbsp;

백엔드 엔지니어에게 신규 기능 개발보다 수십 배 더 끔찍한 작업은 단연 '레거시 마이그레이션(Legacy Migration)'이다. 10년 된 거대한 Monolithic 서버의 낡은 Java 코드를 최신 Node.js 기반의 MSA로 전환하는 프로젝트. 문서는 없고, 당시 개발자는 모두 퇴사했으며, 코드 자체가 유일한 명세서인 상황이다.

&nbsp;

가장 큰 공포는 "내가 새로 짠 API가 기존 레거시 API와 100% 동일한 결과를 반환하는가?"를 보장할 수 없다는 점이다. 수만 가지의 엣지 케이스(Edge Case)를 단위 테스트로 커버하는 것은 불가능하다. 본 글에서는 Envoy Proxy를 활용하여 라이브 트래픽을 몰래 복제(Mirroring)하고, 수백만 건의 JSON 응답 결과를 LLM 기반의 '의미론적 Diff 엔진'으로 자동 검증하여 결함률 0%의 무중단 마이그레이션을 이뤄낸 실전 아키텍처를 상세히 해부한다.

&nbsp;

&nbsp;

&nbsp;

# 1. 아키텍처: 섀도우 트래픽(Shadow Traffic) 라우팅

&nbsp;

운영 서버에서 고객의 요청을 직접 신규 서버로 돌리는 것은 자살 행위다. 우리는 L7 프록시인 Envoy(또는 Istio)의 트래픽 미러링 기능을 활용했다.

&nbsp;

## 1-1. 무사고를 보장하는 Fire and Forget 미러링
사용자의 트래픽이 로드 밸런서를 거치면, 메인 요청은 여전히 안전한 레거시(V1) 서버로 향한다. 
동시에 Envoy는 이 HTTP Request(헤더, 바디 포함)를 완벽하게 복제하여 신규(V2) 서버로 쏜다. 중요한 점은 V2 서버의 응답 결과는 사용자에게 반환되지 않고 버려진다는 점이다(Fire and Forget). V2 서버에서 크래시가 나든 타임아웃이 나든 라이브 서비스에는 단 1%의 영향도 미치지 않는다.

&nbsp;

## 1-2. 응답 캡처 파이프라인
단순히 트래픽을 쏘는 것으로 끝나지 않는다. V1 서버와 V2 서버의 응답(Response JSON)을 비교하기 위해, 양쪽 서버는 자신의 응답 페이로드와 `Request ID`를 비동기 메시지 큐(Kafka)로 전송한다. Kafka 컨슈머(Consumer)는 동일한 `Request ID`를 가진 두 개의 응답을 묶어서 검증 파이프라인으로 넘긴다.

&nbsp;

# 2. 한계 돌파: 기계적 JSON Diff의 실패

&nbsp;

처음에는 단순히 Python의 `DeepDiff` 같은 라이브러리를 써서 두 JSON을 기계적으로 비교(Exact Match)하려 했다. 하지만 이 방식은 수만 건의 False Positive(오탐) 알람을 발생시키며 곧바로 폐기되었다.

&nbsp;

- **순서 문제**: V1은 키 값이 `[A, B, C]` 순서로 나오는데 V2는 `[C, A, B]`로 나온다. (JSON 스펙상 순서는 무관하나 기계적 diff는 에러로 잡는다.)
- **Timestamp 미세 오차**: 응답 객체 내의 `processed_at` 시간이 밀리초(ms) 단위로 다르다.
- **포맷의 차이**: V1은 `1000.0` (Float)을 뱉고, V2는 `1000` (Integer)을 뱉지만 비즈니스 로직상으로는 완전히 동일한 값이다.

&nbsp;

&nbsp;

&nbsp;

# 3. 돌파구: LLM 기반의 의미론적(Semantic) Diff 엔진

&nbsp;

이 지루하고 멍청한 오탐을 걸러내기 위해 LLM을 도입했다. 단순히 텍스트를 비교하는 것이 아니라, **비즈니스 도메인의 룰을 이해하고 의미론적으로 두 응답이 같은지 판별하는 'AI 판사'**를 세운 것이다.

&nbsp;

## 3-1. Validation Prompt 설계
에이전트에게 두 개의 JSON과 함께 '무시해야 할 차이점'에 대한 명확한 룰(Rubric)을 부여한다.

&nbsp;

```python
prompt = f"""
당신은 레거시 마이그레이션 검증을 담당하는 시니어 QA 엔지니어입니다.
동일한 요청에 대해 V1(레거시)과 V2(신규) 서버가 반환한 두 JSON 응답을 비교하십시오.

[검증 규칙]
1. 배열 내 요소의 순서 차이는 무시하십시오.
2. `created_at`, `updated_at` 등 타임스탬프 필드의 1초 이내 오차는 무시하십시오.
3. 데이터 타입의 캐스팅 차이(예: "100" vs 100)는 값이 논리적으로 같다면 무시하십시오.
4. V1에 존재하던 `deprecated_field`가 V2에서 누락된 것은 정상적인 스펙 변경이므로 무시하십시오.

[비교 데이터]
V1 Response: {v1_json}
V2 Response: {v2_json}

[출력 형식]
반드시 아래 JSON 포맷으로 응답하십시오.
{{
  "is_semantically_equal": boolean,
  "critical_differences": ["실제 비즈니스 로직에 영향을 미치는 치명적 차이점 설명"]
}}
"""
```

&nbsp;

## 3-2. 비용 최적화와 샘플링
모든 요청(초당 수백 건)을 LLM에 태우는 것은 파산의 지름길이다.
- **1차 필터링**: 기존의 기계적 `DeepDiff`를 먼저 수행한다. 완벽히 똑같은 80%의 트래픽은 패스시킨다.
- **2차 AI 필터링**: 차이점이 발견된 20%의 응답 중, 동일한 API 엔드포인트와 파라미터 패턴을 가진 요청을 100개 단위로 그룹화(Clustering)하여 대표 샘플 1개씩만 LLM에게 검증을 맡긴다. 이 샘플링 전략으로 토큰 비용을 99% 이상 삭감했다.

# 4. 실전 결과: 결함 제로의 라우팅 스위치

&nbsp;

이 AI 섀도우 트래픽 검증 파이프라인을 2주간 가동하며 V2 서버의 코드를 지속적으로 수정했다. 
초기에는 `is_semantically_equal: false` 비율이 15%에 달했다. V1 서버의 코드가 오랫동안 방치되면서 API 명세서와 달리 이상한 널(Null) 처리를 하고 있었던 엣지 케이스들을 LLM이 정확히 집어낸 덕분이다.

&nbsp;

결함률이 0.01% 미만으로 수렴한 시점, 우리는 Envoy의 라우팅 가중치를 V2 서버로 100% 스위치(Switch)했다. 단 한 건의 CS 클레임이나 장애 없이 거대한 레거시 서버를 영원히 셧다운(Shutdown)시키는, 백엔드 엔지니어 인생 최고의 카타르시스를 맛본 순간이었다.

&nbsp;

# 결론: 마이그레이션은 개발이 아니라 검증의 예술이다

&nbsp;

코드를 새로 짜는 것은 주니어 개발자도 할 수 있다. 하지만 새로 짠 코드가 기존 시스템을 완벽하게 대체할 수 있음을 수학적, 데이터적으로 증명해 내는 것은 시스템 아키텍처와 네트워크 인프라, 그리고 데이터 검증 파이프라인을 다룰 줄 아는 시니어 엔지니어만의 고유한 영역이다.

&nbsp;

단순 반복되는 검증 로직에 AI의 추론 능력을 결합하는 순간, 수개월이 걸릴 마이그레이션 프로젝트의 QA 비용과 심리적 압박감은 극적으로 사라진다. 섀도우 트래픽과 LLM의 만남은 레거시 청산의 새로운 패러다임이다.
