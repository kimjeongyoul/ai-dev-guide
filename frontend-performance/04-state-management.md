# [FE 최적화 4편] 상태 관리의 선택 — Zustand와 React Query의 기술적 시너지

&nbsp;

프론트엔드 애플리케이션의 성능 저하 주범 중 하나는 통제되지 않은 리렌더링(Re-rendering)이다. 특히 과거 Redux 중심의 아키텍처에서는 모든 상태를 거대한 단일 스토어(Single Source of Truth)에 밀어 넣으면서, 무관한 상태 변화가 앱 전체의 컴포넌트 트리를 흔드는 현상이 빈번했다. 하나의 모달 창을 닫았을 뿐인데, 옆에 있던 거대한 차트 컴포넌트가 다시 그려지는 식이다.

&nbsp;

현대적인 프론트엔드 최적화의 핵심은 상태의 성격을 **'서버 상태(Server State)'**와 **'클라이언트 UI 상태(Client State)'**로 엄격히 분리하고, 각각에 특화된 경량 구독 모델을 채택하는 것이다. 본 글에서는 Zustand와 React Query가 어떻게 불필요한 연산을 줄이고 데이터 일관성을 확보하여 애플리케이션의 런타임 성능(특히 INP 지표)을 개선하는지 심층적으로 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. 서버 상태의 독립과 자동화: React Query (TanStack Query)

&nbsp;

우리가 클라이언트에서 다루는 데이터의 90% 이상은 사실 DB에서 가져온 서버 상태다. 이를 전역 상태 저장소에 복사해두고 수동으로 액션과 리듀서를 만들어 관리하는 순간, '데이터 동기화'라는 지옥문이 열린다.

&nbsp;

## 1-1. SWR (Stale-While-Revalidate) 전략
React Query의 핵심 철학은 "캐시된 데이터는 항상 낡은(Stale) 상태가 될 수 있다"는 사실을 인정하는 데서 출발한다.
- 사용자가 컴포넌트를 방문하면, 먼저 메모리에 캐시된 기존 데이터를 즉각적으로 보여준다 (로딩 스피너 없음).
- 동시에 백그라운드에서 조용히 API를 재호출(Revalidate)하여 신선한 데이터를 가져온다.
- 데이터가 변경되었을 때만 컴포넌트를 리렌더링하여 화면을 업데이트한다.
이러한 패턴은 사용자 체감 대기 시간(Perceived Latency)을 0에 가깝게 만들며 UX를 획기적으로 향상시킨다.

&nbsp;

## 1-2. Deduping과 Batching
프론트엔드 아키텍처가 컴포넌트 단위로 파편화되면서, 동일한 API(예: 로그인한 유저 정보 조회)를 헤더, 사이드바, 프로필 영역에서 각각 호출하는 문제가 발생한다.
- **요청 병합(Deduping)**: React Query는 짧은 시간(기본 수 밀리초) 내에 발생하는 동일한 쿼리 키(Query Key)의 요청을 단일 네트워크 호출로 병합한다. 3번 호출해도 실제 서버로 가는 요청은 1번이다.
- **일괄 업데이트(Batching)**: API 응답이 도착하여 상태가 변할 때, 리액트의 라이프사이클에 맞춰 업데이트를 한 번에 묶어서(Batch) 처리하므로 불필요한 중간 리렌더링을 차단한다.

# 2. 클라이언트 상태의 미니멀리즘: Zustand

&nbsp;

서버 데이터를 덜어내고 나면, 프론트엔드에는 다크모드 여부, 사이드바 개폐 상태, 복잡한 폼의 임시 저장 값 등 순수하게 클라이언트에서만 생명주기를 갖는 UI 상태만 남게 된다. 이 영역에서 Zustand는 압도적인 가벼움과 렌더링 통제력으로 대세가 되었다.

&nbsp;

## 2-1. 선택적 구독 (Selector Optimization)을 통한 렌더링 차단
Zustand는 내부적으로 발행/구독(Pub/Sub) 모델과 클로저를 사용하여 리액트 컴포넌트 트리 외부에 상태를 보관한다. 리액트의 Context API는 상위에서 값이 변하면 해당 Context를 구독하는 하위 트리가 무조건 리렌더링되는 치명적인 성능 결함이 있다.

&nbsp;

Zustand는 Selector 패턴을 통해 이를 우회한다.
```javascript
// 상태 스토어 정의
const useStore = create((set) => ({
  bears: 0,
  fishes: 0,
  increaseBears: () => set((state) => ({ bears: state.bears + 1 })),
}));

// 컴포넌트에서의 구독
// 오직 state.bears 값이 변경될 때만 이 컴포넌트가 리렌더링됨
const bears = useStore((state) => state.bears); 
```
위와 같이 명시적으로 `bears`만 구독한 컴포넌트는 `fishes`의 값이 아무리 바뀌어도 렌더링되지 않는다. 이는 대규모 폼(Form)이나 복잡한 대시보드 화면에서 메인 스레드 부하를 극적으로 낮추는 핵심 기술이다.

&nbsp;

## 2-2. 미들웨어 생태계와 Persist
단 1KB 내외의 라이브러리임에도 기능적 확장이 유연하다. `persist` 미들웨어를 사용하면 특정 상태를 LocalStorage나 SessionStorage와 자동으로 동기화할 수 있다. 초기 로딩 시 스토리지에서 값을 읽어와 하이드레이션(Hydration)하는 과정이 코드 한 줄로 처리되며, 사용자가 브라우저를 껐다 켜도 작업 상태가 유지되는 매끄러운 경험을 제공한다.

# 3. 실전 아키텍처: 데이터 흐름의 경계 설계

&nbsp;

도구가 아무리 좋아도 아키텍처 경계를 잘못 지으면 성능은 다시 추락한다. 

&nbsp;

1. **글로벌 서버 데이터**: **React Query**. 전역 UI 스토어에 API 응답 데이터를 옮겨 담는 행위(Syncing) 자체를 엄격히 금지해야 한다. 데이터의 진실의 원천(Source of Truth)은 오직 React Query의 QueryCache여야 한다.
2. **글로벌 UI 상태**: **Zustand**. 여러 라우트나 컴포넌트 트리에 걸쳐 공유되는 테마, 알림 메시지(Toast), 전역 모달 상태 등에만 제한적으로 사용한다.
3. **로컬 상태**: **useState**. 해당 컴포넌트와 그 직계 자식들만 알면 되는 상태는 굳이 밖으로 꺼내지 않는다. 전역 상태로 올리는 순간 구독 관리 오버헤드가 발생한다.
4. **URL 상태**: 검색 필터, 정렬 기준 등은 **URL Query Parameter**에 의존해야 한다. 이는 브라우저의 뒤로가기 대응과 링크 공유 가능성을 완벽하게 보장하는 동시에 전역 상태의 비대화를 막는 가장 세련된 방법이다.

# 4. 결론: 덜어낼수록 빨라진다

&nbsp;

과거에는 "어떻게 하면 상태를 중앙에서 강력하게 통제할까?"를 고민했다면, 이제는 "어떻게 하면 상태를 최소한으로 유지하고 책임을 위임할까?"를 고민해야 한다. 

&nbsp;

서버 데이터의 라이프사이클은 전문 관리자인 React Query에게 온전히 위임하고, 클라이언트 상태는 Zustand를 통해 리렌더링 전파를 원천 차단하는 이분법적 설계. 이 단순하고도 명확한 경계 짓기가 복잡한 프론트엔드 프로젝트를 성능 저하 없이 수년간 유지보수 가능하게 만드는 진정한 엔지니어링 전략이다.
