# [FE 최적화 2편] Next.js App Router — 서버 컴포넌트와 스트리밍 아키텍처의 성능학

&nbsp;

Next.js 13부터 도입된 App Router는 단순한 디렉토리 구조의 변경이 아니다. 이는 프론트엔드 애플리케이션이 브라우저와 통신하고 렌더링 리소스를 소비하는 '물리적 경계'를 재정의한 패러다임 시프트다. 핵심은 React Server Components(RSC) 아키텍처의 전면 도입에 있다. 

&nbsp;

기존 Pages Router 생태계가 안고 있던 고질적인 문제들—클라이언트 사이드 자바스크립트 번들의 비대화, 데이터 페칭의 워터폴(Waterfall) 현상, 브라우저 메인 스레드의 렌더링 과부하—를 RSC가 어떻게 구조적으로 해결하는지, 그리고 이 과정에서 발생하는 기술적 복잡성과 트레이드 오프는 무엇인지 프로덕션 레벨의 시각에서 심층 분석한다.

&nbsp;

# 1. 서버 컴포넌트의 기술적 실체: RSC Payload와 Flight Data

&nbsp;

서버 컴포넌트는 서버에서 HTML을 렌더링하고 끝나는 전통적인 SSR(Server Side Rendering)의 개념을 넘어선다. 서버 컴포넌트의 실행 결과는 브라우저가 직접 파싱할 수 있는 특수한 바이너리 포맷인 **RSC Payload (내부적으로 Flight Data라 불림)**로 직렬화(Serialization)되어 클라이언트로 전송된다.

&nbsp;

## 1-1. 번들 사이즈 '0'의 마법과 파싱 비용 제거
클라이언트 컴포넌트(Client Component)는 실행을 위해 리액트 코드와 의존성 라이브러리가 브라우저로 모두 전송되어야 한다. 반면, 서버 컴포넌트는 오직 서버에서만 실행된다.
예를 들어, 무거운 마크다운 파서(`marked`, 약 100KB)나 날짜 포맷팅 라이브러리(`moment`, 약 200KB)를 사용하여 데이터를 가공할 때, 이 작업이 서버 컴포넌트 내에서 이루어진다면 해당 라이브러리의 코드는 브라우저로 단 1바이트도 전송되지 않는다. 브라우저는 라이브러리 코드가 아닌, 파싱이 완료된 '결과물 텍스트'가 포함된 RSC Payload만 받게 된다. 
이는 모바일 기기에서의 자바스크립트 다운로드, 파싱, 컴파일 시간을 극적으로 단축시켜 첫 화면 렌더링 속도(FCP)를 비약적으로 끌어올린다.

&nbsp;

## 1-2. RSC Payload의 직렬화 구조 분석
RSC Payload는 리액트 컴포넌트 트리를 표현하는 고유의 데이터 구조를 가진다. 
- **서버 컴포넌트의 결과**: HTML 노드와 텍스트 데이터가 포함된다.
- **클라이언트 컴포넌트의 참조**: 서버 컴포넌트 트리 내부에 클라이언트 컴포넌트가 존재할 경우, 해당 컴포넌트의 실제 코드를 담는 대신 브라우저가 나중에 JS 번들을 로드할 수 있도록 모듈의 참조 정보(Reference ID)와 전달할 `props` 데이터만 직렬화하여 담는다.

&nbsp;

이러한 분리 구조 덕분에 브라우저는 RSC Payload를 점진적으로 읽어 들이면서 서버에서 만들어진 UI를 DOM에 즉시 반영하고, 클라이언트 컴포넌트만 선택적으로 하이드레이션(Hydration)하여 인터랙션을 부여할 수 있다.

# 2. 스트리밍(Streaming) 렌더링과 가상 TTFB의 극대화

&nbsp;

전통적인 SSR 아키텍처(예: `getServerSideProps`)는 서버에서 페이지에 필요한 모든 데이터를 DB나 외부 API로부터 페칭할 때까지 브라우저에 응답을 멈춘 채 대기한다. 만약 5개의 API 중 하나라도 응답에 3초가 걸린다면, 사용자는 3초 동안 하얀 백지 화면(White Screen of Death)만 보게 된다. 이는 이탈률 증가의 가장 큰 원인이다.

&nbsp;

## 2-1. HTTP/2 스트리밍 프로토콜의 활용
App Router는 Node.js의 `renderToReadableStream` API와 HTTP/2의 스트리밍 기능을 적극 활용한다. 
리액트 트리에서 데이터 의존성이 없는 정적인 부분(예: 헤더, GNB, 사이드바)과 렌더링이 완료된 서버 컴포넌트 조각들을 먼저 HTML 스트림으로 흘려보낸다. 브라우저는 이 스트림을 받는 즉시 화면에 페인팅을 시작한다.

&nbsp;

## 2-2. Suspense 바운더리와 데이터 페칭 워터폴 해소
데이터 로딩이 오래 걸리는 무거운 차트나 상세 리뷰 목록 등은 `<Suspense>` 바운더리로 감싼다. 
서버는 해당 영역의 데이터 페칭을 기다리지 않고 미리 지정된 `fallback`(예: 스켈레톤 UI) HTML을 먼저 전송한다. 이후 백그라운드에서 데이터 페칭과 컴포넌트 렌더링이 완료되면, 그 결과물과 함께 기존의 `fallback` UI를 실제 컴포넌트로 교체하는 아주 작은 인라인 자바스크립트 조각을 스트림의 끝부분에 밀어 넣는다.
이러한 아키텍처는 데이터 로딩에 병목이 발생하더라도 페이지의 TTFB(Time to First Byte)를 밀리초(ms) 단위로 유지할 수 있게 하며, 사용자에게 "애플리케이션이 즉각적으로 반응하고 있다"는 강력한 시각적 피드백(Perceived Performance)을 제공한다.

# 3. 4단계 캐싱 아키텍처 (Next.js Cache Matrix)

&nbsp;

성능 극대화를 위해 Next.js App Router는 브라우저부터 서버까지 아우르는 4단계의 공격적인 캐싱 레이어를 도입했다. 이를 정확히 이해하고 제어하지 못하면 낡은(Stale) 데이터가 노출되는 버그에 시달리게 된다.

&nbsp;

1. **Request Memoization (서버 렌더링 중)**: 리액트 컴포넌트 트리 여러 곳에서 동일한 API 엔드포인트를 호출하더라도(예: `fetch('https://api/user')`), 서버 렌더링 생명주기 내에서는 단 한 번만 실제 네트워크 요청이 발생한다.
2. **Data Cache (서버 지속성)**: `fetch` 함수의 결과를 서버의 메모리나 파일 시스템에 영구적으로 보관한다. 외부 API가 느리더라도 서버는 캐시된 데이터를 즉시 반환하며, `revalidate` 옵션을 통해 백그라운드 무효화(ISR)를 수행할 수 있다.
3. **Full Route Cache (서버 렌더링 결과)**: 빌드 타임이나 런타임에 렌더링이 완료된 RSC Payload와 HTML 자체를 통째로 서버 Edge에 캐싱한다. 정적 페이지의 경우 서버는 별도의 연산 없이 HTML 파일만 서빙하는 수준의 압도적인 속도를 보여준다.
4. **Router Cache (클라이언트 메모리)**: 사용자가 애플리케이션을 탐색할 때, 브라우저의 메모리에 방문했던 페이지의 RSC Payload를 캐싱한다. 뒤로 가기나 재방문 시 서버에 요청을 보내지 않고 0ms에 가까운 화면 전환을 구현한다.

# 4. 실무 아키텍처: Client Boundary의 설계와 트레이드 오프

&nbsp;

가장 뛰어난 최적화는 서버 컴포넌트를 최대한 넓게 유지하고, 클라이언트 컴포넌트를 최소화하는 것이다. 하지만 실무에서는 이 둘의 경계(Boundary)를 짓는 과정에서 딜레마가 발생한다.

&nbsp;

## 4-1. Leaf Component 패턴으로 인터랙션 격리
`'use client'` 지시어는 감염성(Infection)이 있다. 부모 컴포넌트가 클라이언트 컴포넌트가 되면, 그 자식 컴포넌트들도 모두 클라이언트에서 렌더링 및 번들링 대상이 된다. 
따라서 상태 상태(`useState`)나 이벤트 리스너(`onClick`)가 필요한 UI 요소(예: 찜하기 버튼, 토글 스위치)는 트리의 최하단(Leaf)으로 분리하여 쪼개야 한다.

&nbsp;

```tsx
// ❌ Bad Pattern: 페이지 전체가 클라이언트 번들로 전송됨
'use client';
export default function Page() {
  const [isOpen, setIsOpen] = useState(false);
  // 무거운 데이터와 마크다운 파서가 포함된 거대 로직
  return (
    <div>
      <HeavyMarkdown content={data.content} />
      <button onClick={() => setIsOpen(true)}>Open</button>
    </div>
  );
}

// ✅ Good Pattern: 상태가 필요한 버튼만 클라이언트 컴포넌트로 분리
// page.tsx (Server Component)
export default function Page() {
  return (
    <div>
      <HeavyMarkdown content={data.content} />
      <ClientToggleButton /> {/* 이 내부만 'use client' 선언 */}
    </div>
  );
}
```

&nbsp;

## 4-2. 직렬화(Serialization) 제약과 데이터 전달
서버 컴포넌트에서 클라이언트 컴포넌트로 데이터를 `props`로 넘길 때는 반드시 JSON 직렬화가 가능한 타입이어야 한다. Date 객체, 함수, 클래스 인스턴스 등은 전달 과정에서 에러를 뱉는다. 이 제약은 종종 불편함을 유발하지만, 반대로 거대한 객체나 콜백 함수를 무분별하게 넘기는 안티 패턴을 원천적으로 차단하여 번들 크기를 제어하는 긍정적인 역할도 수행한다.

# 5. 결론: 프론트엔드 엔지니어링의 중심 이동

&nbsp;

Next.js App Router와 RSC의 도입은 단순히 더 빠른 화면 렌더링을 의미하지 않는다. 이는 프론트엔드 개발자가 '브라우저'라는 한정된 샌드박스를 벗어나, 서버의 컴퓨팅 파워와 캐시 아키텍처를 능동적으로 설계해야 함을 의미한다. 

&nbsp;

클라이언트의 브라우저에서 수행하던 데이터 페칭과 컴포넌트 연산을 백엔드 인프라로 이전함으로써, 저사양 기기와 열악한 네트워크 환경을 가진 전 세계 사용자들에게 일관되고 압도적인 로딩 경험을 제공할 수 있게 되었다. 서버와 클라이언트의 경계를 완벽하게 제어하는 능력, 그것이 모던 프론트엔드 성능 최적화의 핵심이다.
