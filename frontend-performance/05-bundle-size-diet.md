# [FE 최적화 5편] 번들 사이즈 다이어트 — 트리 쉐이킹과 코드 스플리팅 실전

&nbsp;

프론트엔드 성능 최적화의 모든 노력은 결국 '브라우저로 전송되는 자바스크립트의 절대적인 양'을 줄이는 귀결점으로 향한다. 
아무리 서버가 응답을 빨리 주고, 이미지를 최적화하고, 렌더링 로직을 정교하게 다듬어도, 초기 번들 사이즈가 수 메가바이트(MB)를 넘어가는 순간 저사양 모바일 기기에서의 TBT(Total Blocking Time)와 초기 로딩 지연은 피할 수 없다. 코드를 다운로드하는 데 걸리는 시간보다, 이를 브라우저 엔진이 파싱(Parsing)하고 컴파일(Compilation)하며 실행하는 CPU 연산 비용이 훨씬 치명적이기 때문이다.

&nbsp;

단순히 기능을 추가하는 방어적 코딩을 넘어, 불필요한 코드를 도려내는 **트리 쉐이킹(Tree Shaking)**과 리소스 로딩 시점을 영리하게 분산하는 **코드 스플리팅(Code Splitting)**의 기술적 원리와 런타임 성능 확보 전략을 심층적으로 다룬다.

&nbsp;

&nbsp;

&nbsp;

# 1. 트리 쉐이킹(Tree Shaking): 죽은 코드의 정적 분석과 제거

&nbsp;

트리 쉐이킹은 나무를 흔들어 썩은 잎사귀를 떨어뜨리듯, 애플리케이션에서 실제로 호출되지 않는 '죽은 코드(Dead Code)'를 빌드 결과물에서 제외하는 최적화 기법이다. Webpack, Rollup, Vite 등 모던 번들러가 이를 지원하지만, 개발자의 구조적 이해가 동반되지 않으면 전혀 작동하지 않는 까다로운 기능이다.

&nbsp;

## 1-1. ES Modules (ESM)의 강제성
트리 쉐이킹이 작동하기 위한 대전제는 의존성 관계가 런타임이 아닌 빌드 타임에 확정되어야 한다는 것이다.
- `import`와 `export`를 사용하는 ESM 체계에서는 모듈의 구조가 정적(Static)이므로, 번들러가 AST(Abstract Syntax Tree)를 순회하며 어떤 함수가 쓰이고 버려졌는지 100% 추적할 수 있다.
- 반면, Node.js 생태계에서 파생된 CommonJS(`require`)나 동적 조건문 안에서 모듈을 불러오는 방식은 런타임에 분기가 결정되므로 번들러가 보수적으로 코드를 모두 남겨두어 최적화가 실패한다. 라이브러리 도입 시 패키지 레지스트리에서 ESM 지원 여부를 가장 먼저 확인해야 한다.

&nbsp;

## 1-2. Side Effects의 공포와 통제
빌드 도구는 특정 코드를 지웠을 때 외부 스코프의 변수를 변경하거나 프로토타입을 오염시키는 '부수 효과(Side Effect)'가 발생할지 확신하지 못하면, 안전을 위해 해당 코드를 삭제하지 않는다.
- **package.json 명시**: 패키지 개발자 또는 프로젝트 환경에서 `"sideEffects": false` 설정을 `package.json`에 명시하면, 번들러에게 "우리 코드는 순수하니 쓰이지 않으면 안심하고 날려버려라"라는 명확한 시그널을 주어 트리 쉐이킹 타율을 극도로 높일 수 있다.
- **순수 함수 지향**: 모듈 최상단 스코프에서 직접 코드를 실행하거나 DOM을 조작하는 행위를 금지하고, 모든 부수 효과는 함수 실행 컨텍스트 내부로 격리해야 한다.

# 2. 코드 스플리팅: 청크(Chunk) 분할과 지연 로딩 전략

&nbsp;

트리 쉐이킹으로 필요 없는 코드를 모두 버렸다 하더라도, 남은 '유효한 코드'를 모두 하나의 거대한 `main.js`에 담아 서빙하는 것은 사용자 경험에 대한 직무 유기다. 

&nbsp;

## 2-1. Route Level Splitting (라우트 기반 분할)
사용자가 로그인 페이지에 접속했을 때, 결제 페이지나 어드민 대시보드의 코드를 미리 다운로드할 필요는 없다. 
Next.js와 같은 최신 SSR 프레임워크는 라우트(페이지) 단위로 번들을 자동 분할한다. 사용자가 링크를 호버(Hover)하거나 뷰포트에 링크가 노출되는 시점에 백그라운드에서 해당 페이지의 JS 청크를 미리 가져오는(Prefetching) 영리한 전략과 결합되어, 초기 로딩은 가벼우면서 페이지 전환은 SPA처럼 매끄럽게 유지한다.

&nbsp;

## 2-2. Component Level Splitting (동적 임포트)
페이지 내에서도 당장 화면에 보이지 않는 무거운 요소들이 존재한다.
- **적용 대상**: 사용자 클릭 후 뜨는 거대한 모달 창, 수백 킬로바이트를 차지하는 차트(Chart) 라이브러리, 사용자가 탭을 전환해야 보이는 텍스트 에디터(WYSIWYG) 등.
- **구현**: React의 `lazy`와 `Suspense`, 또는 Next.js의 `dynamic` 임포트를 사용하여 해당 컴포넌트 렌더링 시점에 네트워크 요청을 발생시킨다.
  ```javascript
  import dynamic from 'next/dynamic'

  // 차트 컴포넌트는 클라이언트 렌더링 시점에만 필요한 거대 리소스
  const HeavyDashboardChart = dynamic(() => import('@components/HeavyChart'), {
    ssr: false, // 서버 사이드 렌더링에서 제외시켜 Node.js 서버 부하까지 경감
    loading: () => <SkeletonGraph />
  })
  ```

# 3. 실전 번들 다이어트와 시각화 분석

&nbsp;

최적화의 시작은 현재 상태를 객관적인 수치와 시각 자료로 파악하는 데서 출발한다.

&nbsp;

1. **번들 시각화 도구 활용**: `webpack-bundle-analyzer` 또는 `@rollup/plugin-visualizer`를 빌드 파이프라인에 연동하라. 생성된 트리맵(Treemap) 차트를 보면 `lodash`의 전체 덩어리나 사용하지 않는 거대한 아이콘 팩 덩어리가 전체 번들의 절반 이상을 차지하는 충격적인 장면을 목격할 수 있다.
2. **Barrel Files (`index.ts`)의 함정 주의**: 여러 모듈을 하나의 `index.ts`에서 모아서 다시 내보내는 패턴은 관리가 편하지만, 특정 모듈 하나만 `import`해도 엮여 있는 다른 모듈 파일 전체가 평가(Evaluation)되면서 트리 쉐이킹이 무력화될 수 있다. 깊이가 깊은 모듈은 명시적인 경로로 직수입하는 것이 안전하다.
3. **가성비 교체**: `moment.js` (수백 KB)를 `dayjs`나 `date-fns` (단 몇 KB)로 걷어내고, 복잡한 렌더링 로직이 없는 애니메이션 라이브러리는 가벼운 CSS 트랜지션으로 대체하는 결단력이 필요하다.

# 결론: 비대해진 자바스크립트는 기술 부채의 정점이다

&nbsp;

새로운 npm 패키지를 설치하는 데는 단 5초의 시간과 키보드 타이핑 몇 번이면 족하지만, 그 무거워진 코드가 사용자 기기에서 야기하는 렌더링 지연과 이탈률 증가는 돌이킬 수 없는 비즈니스 손실을 초래한다. 

&nbsp;

번들 사이즈 최적화는 프로젝트 오픈 직전에 몰아서 하는 대청소가 아니다. 아키텍처 설계 단계부터 철저한 모듈 격리를 고민하고, 매 빌드 시 생성되는 자바스크립트 용량 임계치(Budget)를 설정하여 CI 파이프라인에서 감시하는 것. 그것이 진정한 모던 프론트엔드 엔지니어링의 기본기다. 가벼운 코드가 곧 가장 강력한 사용자 경험임을 명심하라.
