# [FE 최적화 10편] 완벽한 접근성(a11y) — 컴퓨터 비전과 LLM을 결합한 동적 ARIA 주입과 접근성 트리(AOM) 최적화

&nbsp;

현대의 웹 성능 최적화는 단순히 페이지가 빨리 뜨는 것(LCP)이나 버벅거리지 않는 것(INP)만을 의미하지 않는다. 어떤 신체적 조건과 입력 장치를 가진 사용자라도 정보의 구조를 완벽하게 파악하고 시스템과 상호작용할 수 있도록 돕는 **접근성(Accessibility, a11y)** 확보야말로, 진정한 의미의 '보편적 성능 최적화'이자 모던 웹 엔지니어링의 궁극적 지향점이다. 

&nbsp;

하지만 현실의 프론트엔드 개발 조직에서 ARIA(Accessible Rich Internet Applications) 속성을 완벽하게 매핑하고 대체 텍스트(`alt`)를 작성하는 작업은 일정이 촉박할 때 가장 먼저 버려지는 '사치' 취급을 받는다. 본 글에서는 브라우저 내부의 접근성 객체 모델(AOM)의 동작 메커니즘을 짚어보고, 컴퓨터 비전 기능이 탑재된 다중 양식(Multimodal) LLM을 파이프라인에 결합하여 이 고통스러운 수동 작업의 한계를 부수고 런타임/빌드 타임에 접근성 속성을 완전 자동화로 주입해 내는 혁신적 아키텍처를 심층 탐구한다.

&nbsp;

&nbsp;

&nbsp;

# 1. 브라우저의 보이지 않는 뼈대: AOM (Accessibility Object Model)

&nbsp;

개발자들은 브라우저가 HTML을 파싱해 DOM 트리를, CSS를 파싱해 CSSOM 트리를 만든다는 사실은 잘 알고 있다. 그러나 스크린 리더(VoiceOver, NVDA 등)가 어떻게 화면을 읽어주는지에 대해서는 무지하다. 스크린 리더는 화면의 픽셀을 읽지 않는다. 

&nbsp;

브라우저는 DOM 트리를 바탕으로 시각적 정보를 제거하고 '의미론적 정보와 상호작용의 상태'만을 담은 또 다른 병렬 구조의 트리, 즉 **AOM(접근성 트리)**을 실시간으로 구축하여 운영체제의 접근성 API로 전달한다.
- 햄버거 메뉴를 의미론 없는 단순 `<div>`로 묶고 클릭 이벤트를 달아놓는다면, DOM 트리는 정상적으로 렌더링되지만 AOM 트리에서는 그 요소가 '버튼'인지, 현재 메뉴가 '열려있는(expanded)' 상태인지 전혀 표현되지 못한다. 
- 스크린 리더 사용자는 그 햄버거 메뉴가 화면에 존재하는지조차 인지할 수 없는 완벽한 정보의 블랙홀에 빠지는 것이다.

&nbsp;

이를 구제하기 위해 고안된 것이 `role="button"`, `aria-expanded="false"` 등의 ARIA 속성이며, 이는 AOM 트리의 비어있는 구멍을 강제로 수선해 주는 필수적인 깁스 역할을 한다.

&nbsp;

# 2. 전통적 한계: 휴먼 에러와 컨텍스트 단절

&nbsp;

접근성을 자동화하려는 시도가 없었던 것은 아니다. `eslint-plugin-jsx-a11y` 같은 린터(Linter) 도구는 이미지에 `alt` 속성이 빠져있거나 클릭 가능한 요소에 키보드 이벤트(onKeyDown)가 누락되면 에러를 뱉는다.

&nbsp;

하지만 정적 분석 도구는 '컨텍스트'를 읽지 못한다.
- 개발자는 린트 에러를 통과하기 위해 이미지에 무의미하게 `alt="image"` 또는 `alt="배너"`라고 대충 채워 넣는다. 이는 스크린 리더 사용자에게 아무런 정보도 주지 못하는 가비지 데이터다.
- 복잡한 모달 창 구조에서 포커스 트랩(Focus Trap)이 올바르게 설계되었는지, 혹은 `aria-labelledby`가 가리키는 ID의 문맥이 실제 모달의 목적과 부합하는지 린터는 판별할 지능이 없다.

# 3. 비전 LLM 아키텍처: 화면의 '맥락'을 코드로 주입하다

&nbsp;

이 절대적 한계를 타개하기 위해, 화면의 시각적 형태(Vision)와 코드의 구조적 형태(Code)를 동시에 이해하는 멀티모달 LLM(GPT-4o Vision, Claude 3.5 Sonnet 등)을 CI/CD 빌드 파이프라인에 결합하는 아키텍처를 도입했다.

&nbsp;

## 3-1. 스크린샷과 AST의 교차 분석 (Cross-modal Parsing)
1. **스냅샷 캡처**: Playwright나 Cypress 같은 E2E 테스트 도구를 활용하여 주요 컴포넌트나 라우트 화면의 고해상도 스크린샷을 찍는다.
2. **코드 트리 추출**: 동시에 해당 화면을 구성하는 React JSX 컴포넌트 트리의 텍스트 데이터를 추출한다.
3. **AI 프롬프트 인젝션**: 비전 LLM에게 이미지와 코드를 동시에 넘기며 '맥락 기반 접근성 보강'을 지시한다.

&nbsp;

```python
prompt = f"""
당신은 최고 수준의 시각 장애인 접근성(WCAG 2.1) 심사관입니다.
제공된 [화면 스크린샷]과 이를 구성하는 [React JSX 코드]를 교차 분석하십시오.

[작업 지시]
1. 화면 상의 시각적 흐름(맥락)을 파악하여, `<img>` 태그에 누락되었거나 성의 없게 작성된 `alt` 속성을 "이미지의 실제 역할과 내포된 텍스트 정보"가 명확히 담기도록 상세하게 재작성하십시오. (예: "배너" -> "2024년 여름 세일 최대 50% 할인 안내 배너")
2. 시각적으로는 탭(Tab) 메뉴로 기능하지만 코드는 단순 리스트(`<ul>`)로 작성된 요소를 찾아내어, 적절한 `role="tablist"`, `role="tab"`, 그리고 활성화 상태를 나타내는 `aria-selected` 속성이 주입되도록 코드를 교정하십시오.
3. 수정된 JSX 코드 전문만을 반환하십시오.
"""
```

&nbsp;

## 3-2. 런타임 동적 주입 (Dynamic Injection)
빌드 타임에 코드를 수정하는 방식 외에도, CMS 등에서 동적으로 올라오는 배너 이미지들의 접근성을 위해 서버 사이드 렌더링(SSR) 타임에 개입할 수도 있다.
백엔드 노드(Node.js) 서버가 HTML 스트림을 내려보내기 직전, 등록된 이미지 URL을 경량 비전 모델에 1회성으로 태워 `alt` 텍스트를 자동 생성하고 이를 캐싱 서버(Redis)에 저장하여 이후 요청부터는 생성된 대체 텍스트가 담긴 HTML을 완벽하게 서빙하는 것이다.

# 4. 결론: 가장 인간적인 기술, 가장 기술적인 해법

&nbsp;

접근성(a11y) 최적화는 소수의 장애인을 위한 배려가 아니라, 브라우저와 운영체제의 의미론적(Semantic) 파이프라인을 얼마나 정확히 이해하고 통제할 수 있는가를 증명하는 프론트엔드 엔지니어링의 최고봉이다. 검색 엔진의 크롤링 봇(Bot) 역시 스크린 리더와 완벽히 똑같은 방식으로 AOM 구조를 해석하기에, 완벽한 접근성은 곧 완벽한 SEO(검색 엔진 최적화)와 비즈니스 트래픽 증가로 이어진다.

&nbsp;

수동으로 키보드 탭 키를 눌러가며 린트 에러를 잡던 고통스러운 노가다의 시대는 저물었다. 화면의 시각적 구조를 수학적으로 이해하는 AI의 컴퓨터 비전과, 코드의 의미론을 교정하는 텍스트 추론 능력을 파이프라인에 결합하는 순간, 우리는 '누구도 배제되지 않는 웹 생태계'라는 원대한 철학을 가장 하드코어한 기술적 자동화로 구현해 내게 된다.

&nbsp;
