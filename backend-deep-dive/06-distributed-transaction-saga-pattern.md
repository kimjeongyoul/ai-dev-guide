# [백엔드 6편] Saga 패턴의 설계와 운영 — 분산 환경에서 데이터 일관성을 어떻게 보장할 것인가?

&nbsp;

모놀리식 아키텍처의 시대에 데이터 정합성은 당연한 권리였다. `@Transactional` 어노테이션 하나면 DB가 알아서 원자성(Atomicity)을 보장해 주었기 때문이다. 하지만 마이크로서비스(MSA)로 넘어온 순간, 이 평화는 깨졌다. '주문', '결제', '재고'가 각자의 DB를 가진 환경에서 하나의 비즈니스 로직은 여러 번의 네트워크 요청으로 쪼개졌고, 그중 하나라도 실패하면 시스템은 '부분 성공'이라는 끔찍한 상태에 빠지게 되었다. 

&nbsp;

전통적인 2PC(2-Phase Commit)는 성능 저하와 가용성 하락으로 인해 MSA에서 사형 선고를 받았다. 본 글에서는 그 대안으로 떠오른 **Saga 패턴**이 어떻게 '실패를 되돌리는 기술(Compensating Transaction)'을 통해 최종 정합성을 실현하는지 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. 분산 트랜잭션의 딜레마: 2PC의 실패와 가용성

&nbsp;

2PC는 모든 참여 노드가 "준비 완료"를 외칠 때까지 자원을 잠그는(Locking) 방식이다. 하지만 분산 환경에서 한 노드라도 응답이 늦어지면 전체 시스템이 마비된다. 가용성이 생명인 현대 웹 서비스에서 이는 치명적이다. 

&nbsp;

엔지니어는 이제 **ACID**라는 완벽한 정합성을 포기하고, **BASE**(Basically Available, Soft state, Eventual consistency) 모델을 받아들여야 한다. Saga 패턴은 바로 이 BASE 철학을 구현하는 구체적인 방법론이다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. Saga의 두 가지 검: Choreography와 Orchestration

&nbsp;

Saga 패턴은 하나의 거대한 트랜잭션을 순차적인 로컬 트랜잭션의 집합으로 정의한다. 핵심은 **'실패 시 보상 트랜잭션 실행'**이다.

&nbsp;

- **Choreography (코레오그래피)**: 중앙 통제자 없이 서비스들이 이벤트를 주고받으며 자율적으로 흐름을 이어간다. 구현이 단순하고 서비스 간 결합도가 낮지만, 비즈니스 흐름이 복잡해지면 전체 구조를 파악하기 어렵다.
- **Orchestration (오케스트레이션)**: 중앙의 'Saga 매니저'가 전체 흐름을 제어한다. 복잡한 비즈니스 로직을 한눈에 파악할 수 있고 상태 관리가 쉽지만, 매니저 자체가 병목이 되거나 단일 장애점(SPOF)이 될 위험이 있다.

&nbsp;

&nbsp;

---

&nbsp;

# 3. 실전 엔지니어링 교훈: 멱등성(Idempotency)은 선택이 아닌 필수다

&nbsp;

Saga 패턴을 도입하려는 시니어 설계자는 다음의 '부작용'에 대비해야 한다.

1. **보상 트랜잭션의 실패**: "결제 취소를 시도했는데 실패하면 어쩌지?"라는 질문에 답해야 한다. 보상 로직은 성공할 때까지 무한 재시도하거나, 사람이 개입하는 '데드 레터 큐(DLQ)' 처리 프로세스가 반드시 필요하다.
2. **멱등성 설계**: 네트워크 장애로 인해 같은 이벤트가 두 번 발행될 수 있다. 모든 로컬 트랜잭션은 동일한 요청을 여러 번 받아도 한 번만 처리되도록 **멱등성(Idempotency)**이 보장되어야 한다.
3. **사용자 경험의 변화**: "주문 완료" 메시지를 보냈지만 0.5초 뒤에 "재고 부족으로 취소되었습니다"라는 알림이 갈 수 있음을 비즈니스 팀과 합의해야 한다.

정합성은 더 이상 DB의 기능이 아니라, **백엔드 아키텍처의 설계 역량**이다. Saga 패턴은 데이터의 완벽함을 조금 양보하는 대신, 거대한 시스템이 멈추지 않고 전진할 수 있는 생명력을 부여한다.
