# [백엔드 9편] Cache Stampede 현상과 방어 전략 — 캐시 만료 순간의 트래픽 폭주를 어떻게 제어할 것인가?

&nbsp;

성능 최적화를 위해 도입한 캐시 레이어는 특정 상황에서 인프라 전체를 마비시키는 위협이 되기도 합니다. 
 특히 대량의 트래픽이 발생하는 서비스에서 캐시가 만료되는 순간, 수많은 요청이 한꺼번에 DB로 몰리는 Cache Stampede 현상은 연쇄적인 장애를 유발합니다. 본 아티클에서는 Cache Stampede의 발생 원리를 살펴보고 이를 방어하기 위한 전략을 분석합니다.

---

## 1. Cache Stampede의 발생 메커니즘

Cache Stampede는 단순한 성능 저하가 아닌 인프라의 임계점을 넘어서는 부하를 발생시킵니다.

1.  캐시 만료: TTL이 설정된 인기 캐시 키가 만료됩니다.
2.  요청 폭증: 동일한 데이터를 요청하는 수많은 스레드가 캐시 미스를 인지합니다.
3.  DB 부하: 모든 스레드가 동시에 DB에 쿼리를 날리고 결과를 캐시에 쓰려고 시도합니다.
4.  시스템 장애: DB 커넥션 풀이 고갈되고 CPU 사용량이 치솟으며 서비스 장애로 이어집니다.

---

## 2. 전략 1: Jitter를 통한 만료 시간 분산

가장 단순하고 효과적인 방법은 모든 캐시의 만료 시간을 조금씩 다르게 설정하는 것입니다. TTL에 미세한 난수를 더해 만료 시점을 분산시키면, 특정 시점에 요청이 몰리는 현상을 방지하여 DB에 가해지는 부하를 낮출 수 있습니다.

---

## 3. 전략 2: PER (Probabilistic Early Recomputation) 알고리즘

PER는 캐시가 실제로 만료되기 전에 확률적으로 미리 갱신을 수행하는 기법입니다. 요청이 들어왔을 때 일정한 수식에 따라 확률적으로 만료 전 갱신을 수행하므로, 만료 시점에 모든 요청이 DB로 몰리는 상황을 차단할 수 있습니다.

---

## 4. 전략 3: 분산 락(Distributed Lock)을 이용한 제어

여러 인스턴스 환경에서 동일한 키에 대한 DB 접근을 하나로 제한하는 방식입니다.

*   구현 방식: 캐시 미스 발생 시 Redis의 SET NX 등을 이용해 락을 획득하고, 락을 얻은 단 하나의 스레드만 DB에서 데이터를 읽어와 캐시를 갱신합니다.
*   트레이드오프: 락을 얻지 못한 다른 스레드들은 대기하거나 이전 버전의 캐시 데이터를 즉시 반환받게 설계할 수 있습니다. 가용성이 중요하다면 락 대기 없이 이전 데이터를 반환하는 방식이 대규모 트래픽 처리에 유리합니다.

---

## 5. 전략 4: 계층적 캐싱 (L1-L2 Hybrid Cache)

애플리케이션 메모리 내 로컬 캐시와 분산 캐시를 병행하는 전략입니다.

*   L1 (Local Cache): 각 서버 노드의 메모리에 위치하며 가장 빠른 응답을 보장합니다.
*   L2 (Remote Cache): 분산 Redis 클러스터로 모든 노드가 데이터를 공유합니다.
*   동작 원리: 먼저 L1을 확인하고 없으면 L2를, 둘 다 없을 때만 DB에 접근합니다. L1의 TTL을 L2보다 짧게 설정하면 안정적인 캐시 갱신 구조를 만들 수 있습니다.

---

## 6. 결론: 서비스 특성에 따른 방어 전략

Cache Stampede 방어는 서비스의 특성에 따라 조합되어야 합니다. 일반적인 정보성 데이터는 Jitter만으로도 효과를 볼 수 있으며, 고부하가 예상되는 핵심 데이터는 PER 알고리즘이나 분산 락을 통한 제어가 효과적입니다. 응답 속도가 중요한 서비스라면 로컬 캐시 조합이 가장 강력한 방어선이 됩니다.
