# [백엔드 9편] Cache Stampede 현상과 방어 전략 — 캐시 만료 순간의 트래픽 폭주를 어떻게 제어할 것인가?

&nbsp;

"어제까지만 해도 멀쩡하던 DB가 왜 갑자기 죽었죠?" 대규모 트래픽을 처리하는 서비스에서 종종 발생하는 미스터리다. 성능을 위해 도입한 캐시가 오히려 시스템을 파괴하는 부메랑이 되어 돌아온 것이다. 캐시가 만료되는 아주 찰나의 순간, 수천 개의 요청이 한꺼번에 원본 DB로 몰려드는 이른바 **Cache Stampede(Thundering Herd)** 현상은 연쇄적인 장애(Cascade Failure)를 유발하는 백엔드의 보이지 않는 자객이다. 

&nbsp;

본 글에서는 캐시 만료의 위기를 우아하게 넘기기 위한 고도화된 방어 전략을 상세히 분석한다.

&nbsp;

&nbsp;

&nbsp;

# 1. Cache Stampede의 메커니즘: 복구 불가능한 부하의 시작

&nbsp;

캐시는 만료되는 즉시 데이터가 사라진다. 이때 들어온 모든 요청은 동시에 "캐시 미스(Cache Miss)"를 인지한다. 수천 개의 스레드가 각자 데이터를 새로 생성하거나 DB에서 읽어오려 시도하면 다음과 같은 사태가 벌어진다.
- **DB CPU 폭주**: 동일한 쿼리가 수천 개 실행되어 DB가 마비된다.
- **중복 연산**: 비싼 연산 로직이 여러 번 중복 실행되어 리소스를 낭비한다.
- **결과**: DB가 느려지면 애플리케이션 스레드가 점유되어 전체 서비스가 먹통이 된다.

# 2. 실전 방어 전략: 찰나를 다스리는 기술

&nbsp;

## 2-1. PER (Probabilistic Early Recomputation) 알고리즘
캐시가 만료되기 전, **확률적**으로 누군가 미리 캐시를 갱신하게 만드는 영리한 방식이다. 캐시 수명의 80% 시점부터 특정 확률로 갱신 로직을 실행하도록 설계한다. 트래픽이 많을수록 누군가 '미리' 갱신할 확률이 높아져 실제 만료 시점에는 데이터가 이미 신선하게 유지된다.

## 2-2. Mutex (분산 락) 활용
캐시 미스가 발생했을 때, 오직 **단 하나의 요청**만 DB에 접근할 수 있도록 락을 거는 방식이다. 나머지 요청들은 잠시 대기하다가 먼저 성공한 놈이 채워넣은 캐시를 읽어간다. 리소스를 가장 완벽하게 보호하지만, 락 대기 시간으로 인한 지연이 발생할 수 있다.

## 2-3. 백그라운드 갱신 (Soft Expiration)
캐시 데이터에 '진짜 만료 시간' 외에 '논리적 만료 시간'을 함께 저장한다. 논리적 만료가 지나면 우선 과거 데이터를 사용자에게 주고, 백그라운드에서 비동기로 캐시를 갱신한다. 사용자 경험 측면에서 가장 부드러운 방식이다.

# 3. 실전 엔지니어링 교훈: 정적 설계보다 동적 대응

&nbsp;

시니어 설계자는 캐시 TTL(Time-To-Live) 설정 하나에도 철학을 담아야 한다.

1. **지터(Jitter)의 도입**: 대량의 캐시가 동시에 만료되지 않도록 TTL에 무작위 값을 더하라. (예: 60분 + `rand(0~5)`분). 이 작은 조치가 Stampede를 분산시키는 최고의 묘수가 된다.
2. **Warm-up 파이프라인**: 서비스 배포 직후나 피크 타임 직전에 주요 데이터를 미리 캐싱하는 '웜업' 프로세스를 자동화하라.
3. **Fallback 설계**: DB가 정말 죽었을 때를 대비해, 조금 낡은 데이터라도 보여줄 수 있는 'Stale-while-revalidate' 전략을 적극 도입해야 한다.

캐싱은 단순히 속도를 높이는 도구가 아니라, **인프라의 생존권을 지키는 방어선**이다. 찰나의 순간을 통제하는 자만이 대규모 트래픽의 파도 위에서 유유히 항해할 수 있다.
