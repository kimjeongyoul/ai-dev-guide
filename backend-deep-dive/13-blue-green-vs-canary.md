# [백엔드 13편] 무중단 배포 전략 — Blue-Green과 Canary 중 당신의 서비스에 맞는 최적의 선택은?

&nbsp;

"배포는 축제가 되어야지, 장례식이 되어서는 안 된다." 넷플릭스나 아마존 같은 거대 IT 기업들이 하루에도 수천 번씩 배포를 진행하면서도 서비스 중단이 없는 비결은 무엇일까? 과거에는 점검 공지를 올리고 서버를 내리는 것이 당연했지만, 현대의 백엔드 아키텍처에서 **Zero-Downtime Deployment**는 선택이 아닌 생존의 문제다.

&nbsp;

본 글에서는 리스크를 관리하는 철학이 완전히 다른 두 가지 배포 전략, **Blue-Green**과 **Canary**의 메커니즘과 그 이면에 숨겨진 트레이드오프를 심층 비교한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. Blue-Green 배포: 완벽한 격리와 즉각적인 롤백

&nbsp;

Blue-Green 전략의 핵심은 **"동일한 규모의 환경을 하나 더 갖는 것"**이다. 현재 운영 중인 환경(Blue)과 똑같은 새로운 환경(Green)을 구성하고, 모든 테스트가 완료되면 로드밸런서의 타겟을 한순간에 Green으로 전환한다.

&nbsp;

- **강점 (The Atomic Switch)**: 전환이 원자적으로 일어나며, 신규 버전에서 치명적인 버그가 발견되어도 로드밸런서 설정만 다시 Blue로 돌리면 즉시 롤백이 가능하다. 배포 과정에서 구버전과 신버전이 공존하는 혼란이 없다.
- **약점 (The Infrastructure Tax)**: 전체 인프라 자원이 일시적으로 2배 필요하다. 비용 문제뿐만 아니라, DB 스키마 변경 시 구버전과 신버전이 같은 DB를 바라볼 때 발생하는 호환성 문제(Backward Compatibility)를 해결하는 것이 가장 큰 숙제다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. Canary 배포: 점진적 검증과 장애 영향도의 최소화

&nbsp;

광산의 유독 가스를 감지하기 위해 카나리아를 먼저 보냈던 것처럼, **Canary 배포**는 트래픽의 아주 일부(예: 5%)만 신규 버전으로 보내 실제 운영 환경에서의 안정성을 모니터링한다.

&nbsp;

- **강점 (Blast Radius Control)**: 아무리 완벽한 테스트 환경도 실제 사용자의 예측 불가능한 트래픽을 재현할 수는 없다. Canary는 실제 사용자 일부에게만 영향을 주며 리스크를 점진적으로 해소한다. 이 과정에서 에러율, Latency, CPU 사용량 등의 지표를 실시간으로 비교 분석한다.
- **약점 (The Complexity of Coexistence)**: 장시간 동안 서로 다른 두 버전이 공존해야 한다. 사용자가 세션 유지 중에 다른 버전의 서버로 연결되어도 문제가 없도록 'Sticky Session'이나 데이터 정합성 관리에 세심한 주의가 필요하다.

&nbsp;

&nbsp;

&nbsp;

# 3. 실전 엔지니어링 교훈: 무엇이 당신의 밤잠을 설칠 것인가?

&nbsp;

전략의 선택은 기술적 선호도가 아닌 **'리스크의 성격'**에 따라야 한다.

1. **Blue-Green**은 인프라 비용이 들더라도 **전환의 확실성**과 **빠른 롤백**이 최우선일 때 적합하다. 특히 배포 빈도가 낮고 한 번의 배포가 대규모일 때 유리하다.
2. **Canary**는 배포 빈도가 높고, 신규 로직이 실제 트래픽 환경에서 어떻게 동작할지 확신할 수 없을 때 선택한다. 이는 **'프로그레시브 딜리버리(Progressive Delivery)'**의 핵심이며, 성숙한 관찰성(Observability) 도구가 뒷받침될 때 비로소 그 진가를 발휘한다.

시니어 엔지니어라면 단순히 배포 도구를 다루는 것을 넘어, **"배포 중 장애 발생 시 우리 팀은 몇 분 안에 감지하고 몇 초 안에 복구할 수 있는가?"**라는 질문에 답할 수 있는 파이프라인을 설계해야 한다.
