# [백엔드 13편] 무중단 배포 전략 — Blue-Green과 Canary 중 당신의 서비스에 맞는 최적의 선택은?

무중단배포, BlueGreen, Canary, 리스크관리, 배포전략, 롤백, 백엔드설계

&nbsp;

Zero-Downtime Deployment는 현대 백엔드 운영의 기본입니다. 하지만 모든 서비스에 동일한 방식이 정답은 아닙니다. 리스크 관리 방식이 전혀 다른 **Blue-Green**과 **Canary** 배포의 트레이드오프를 분석합니다.

&nbsp;

---

&nbsp;

# 1. Blue-Green 배포: 신속한 교체와 즉각적인 롤백

&nbsp;

- **메커니즘**: 현재 버전(Blue)과 동일한 규모의 새로운 환경(Green)을 미리 구성하고, 로드밸런서의 타겟을 한 번에 전환합니다.
- **강점**: 전환이 매우 깔끔하며, 치명적인 오류 발견 시 Blue로 다시 돌리는 롤백 시간이 0에 가깝습니다.
- **약점**: 전체 인프라 자원이 일시적으로 2배 필요하므로 비용 부담이 큽니다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. Canary 배포: 점진적 검증과 장애 영향도 최소화

&nbsp;

- **메커니즘**: 트래픽의 아주 일부(예: 5%)만 신규 버전으로 보내 실제 운영 환경에서의 안정성을 모니터링합니다. 점진적으로 비중을 높여 100%에 도달합니다.
- **강점**: 새로운 코드의 버그나 사용자 반응을 실제 트래픽으로 검증할 수 있어 '장애 영향 범위(Blast Radius)'를 최소화합니다.
- **약점**: 배포 프로세스가 길어지고, 구 버전과 신 버전이 공존하는 동안의 API 하위 호환성 관리가 복잡합니다.

&nbsp;

&nbsp;

---

&nbsp;

# 3. 실전 엔지니어링 교훈: 선택의 기준

&nbsp;

- **Blue-Green**: 인프라 비용보다 **빠른 버전 교체**와 **완벽한 환경 분리**가 중요할 때 선택합니다.
- **Canary**: 신규 기능의 로직이 복잡하여 **실제 사용자 데이터 기반의 검증**이 필수적인 대규모 서비스에서 선택합니다.

최근에는 '프로그레시브 딜리버리(Progressive Delivery)' 관점에서 Canary 배포가 선호되지만, 이는 성숙한 관찰성(Observability) 도구가 뒷받침될 때 비로소 그 진가를 발휘합니다.
