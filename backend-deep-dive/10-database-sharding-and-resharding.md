# [백엔드 10편] DB 샤딩과 리샤딩 전략 — 대규모 데이터 처리를 위한 수평 확장의 기술적 도전 과제는?

DB샤딩, 리샤딩, 수평확장, 샤드키선정, 데이터분산, 대규모데이터, 백엔드설계

&nbsp;

데이터베이스의 수직 확장이 한계에 다다랐을 때, 수평 확장을 구현하는 가장 강력한 방법은 샤딩(Sharding)입니다. 
 하지만 샤딩은 도입하는 순간 조인(Join)의 제약, 트랜잭션 범위 한계, 데이터 불균형 등 막대한 운영 비용이 발생합니다. 본 아티클에서는 성공적인 샤딩을 위한 샤드 키 선정 기법과 서비스 중단 없는 리샤딩 전략을 분석합니다.

---

## 1. 샤드 키 선정 기법

샤드 키는 데이터가 어떤 샤드에 저장될지 결정하는 기준입니다. 잘못된 키 선정은 특정 샤드에만 데이터가 몰리는 Hotspot 현상을 유발합니다.

*   카디널리티(Cardinality): 키값의 종류가 충분히 많아야 합니다. (예: 사용자 ID 등)
*   분포도(Distribution): 시간이 지나도 데이터가 특정 샤드에 치우치지 않고 균등하게 쌓여야 합니다.
*   조회 패턴(Query Pattern): 가장 빈번하게 발생하는 쿼리의 조건절에 포함된 컬럼이어야 샤딩의 성능적 이점을 취할 수 있습니다.

---

## 2. 컨시스턴트 해싱과 데이터 불균형 해소

전통적인 모듈러 방식의 샤딩은 노드가 추가될 때마다 거의 모든 데이터를 재배치해야 하는 단점이 있습니다.

### 2.1 컨시스턴트 해싱 (Consistent Hashing)
데이터와 노드를 가상의 링 위에 배치하고, 각 데이터는 링 위에서 시계 방향으로 가장 가까운 노드에 할당합니다. 노드가 추가되거나 삭제될 때 인접한 노드 사이의 데이터만 이동하면 되므로 재배치 범위를 획기적으로 줄일 수 있습니다.

### 2.2 가상 노드 (Virtual Nodes)
물리적인 노드 하나가 링 위의 여러 지점에 위치하도록 가상 노드를 생성합니다. 이를 통해 각 노드가 담당하는 영역을 세밀하게 쪼갤 수 있으며 데이터 불균형 현상을 완화할 수 있습니다.

---

## 3. 무중단 리샤딩 (Zero-downtime Re-sharding) 전략

운영 중인 서비스의 데이터베이스를 확장하는 과정은 정교하게 설계되어야 합니다.

1.  이중 쓰기 (Dual Write): 애플리케이션 레벨에서 기존 샤드와 신규 샤드 모두에 실시간 데이터를 동시에 씁니다.
2.  데이터 이관 (Migration): 이중 쓰기 시작 이전의 과거 데이터를 백그라운드에서 신규 샤드로 복사합니다. 이때 데이터 충돌은 최신성을 기준으로 보정합니다.
3.  정합성 검증 (Verification): 기존 샤드와 신규 샤드의 데이터 정합성을 체크합니다.
4.  전환 (Cut-over): 읽기 비중을 서서히 신규 샤드로 옮기고 기존 샤드의 불필요한 데이터를 정리합니다.

---

## 4. 샤딩 도입 전 고려해야 할 대안들

샤딩은 운영 복잡도가 매우 높으므로 도입 전 다음과 같은 대안들을 먼저 검토해야 합니다.

*   읽기 전용 복제본(Read-Replica) 확장: 읽기 부하가 문제라면 복제본 추가로 해결 가능한지 검토합니다.
*   캐시 레이어 고도화: 캐시 히트율을 높여 DB 접근 자체를 줄입니다.
*   데이터 아카이빙: 오래된 데이터를 별도 스토리지로 분리하여 활성 데이터셋 크기를 최적화합니다.
*   수직 파티칭 (Vertical Partitioning): 테이블 컬럼을 전략적으로 분리하여 I/O 효율을 높입니다.

---

## 5. 결론: 견고한 데이터 모델 설계를 지향하는 아키텍처

샤딩은 도입 시 막대한 비용이 수반됩니다. 분산 조인의 부재를 극복하기 위한 데이터 비정규화, 분산 트랜잭션을 보완할 Saga 패턴의 결합 등이 함께 고려되어야 합니다. 가장 효율적인 샤딩 전략은 샤딩 도입 시점을 최대한 늦출 수 있도록 데이터 모델을 설계하고 인프라를 최적화하는 것입니다.
