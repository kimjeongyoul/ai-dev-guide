# [백엔드 10편] DB 샤딩과 리샤딩 전략 — 대규모 데이터 처리를 위한 수평 확장의 기술적 도전 과제는?

&nbsp;

데이터베이스의 수직 확장(Scale-up)은 결국 물리적 한계에 부딪힌다. 수억 건의 레코드가 쌓이고 초당 수만 건의 쿼리가 쏟아질 때, 단일 데이터베이스는 비명을 지르며 멈춰 선다. 이때 엔지니어가 꺼내 드는 최후의 카드가 바로 **샤딩(Sharding)**이다. 데이터를 여러 개의 작은 조각(Shard)으로 쪼개어 독립된 서버에 분산 저장하는 기술이다. 

&nbsp;

하지만 샤딩은 '판도라의 상자'와 같다. 도입하는 순간 조인(Join)의 제약, 분산 트랜잭션의 난제, 그리고 무엇보다 고통스러운 **리샤딩(Resharding)**의 위협이 따라온다. 본 글에서는 성공적인 샤딩을 위한 설계 원칙과 무중단 데이터 이사의 메커니즘을 상세히 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. 샤드 키(Shard Key) 선정: 불균형과의 전쟁

&nbsp;

샤딩의 성패는 데이터를 나누는 기준인 **샤드 키** 선정에 달려 있다. 

&nbsp;

- **Hash-based Sharding**: 키값을 해시 함수에 돌려 서버를 결정한다. 데이터가 가장 균등하게 분산되지만, 서버를 추가할 때 해시 결과값이 완전히 바뀌어 모든 데이터를 다시 옮겨야 하는 지옥이 펼쳐진다.
- **Range-based Sharding**: 날짜나 숫자 범위를 기준으로 나눈다. 쿼리가 특정 범위에 몰릴 경우 특정 샤드만 부하를 받는 'Hotspot' 현상이 발생하기 쉽다.

시니어 설계자는 비즈니스 쿼리 패턴을 분석하여, **Write 부하를 균등하게 분산**시키면서도 **대부분의 읽기 요청이 단일 샤드 내에서 해결**되도록 샤드 키를 도출해내야 한다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 리샤딩(Resharding)의 예술: 달리는 자동차의 바퀴 갈아 끼우기

&nbsp;

비즈니스가 성장하여 기존 샤드가 가득 차면, 샤드를 더 쪼개는 '리샤딩'이 필수적이다. 서비스를 멈추지 않고 수 테라바이트의 데이터를 옮기는 것은 고난도 작업이다.

&nbsp;

1. **Consistent Hashing**: 서버가 추가되어도 데이터 이동을 최소화하는 해시 알고리즘을 도입하여 리샤딩의 충격을 줄인다.
2. **Dual Write 전략**: 신규 샤드를 구축하는 동안 애플리케이션에서 구 샤드와 신 샤드 양쪽에 데이터를 동시에 쓴다. 그동안 백그라운드에서 과거 데이터를 마이그레이션하고, 검증이 끝나면 신규 샤드로 트래픽을 완전히 전환한다.

&nbsp;

&nbsp;

---

&nbsp;

# 3. 실전 엔지니어링 교훈: 샤딩은 '최후의 수단'이다

&nbsp;

샤딩은 아키텍처의 복잡도를 수 배로 높이는 무거운 결정이다. 

1. **샤딩 전 최적화**: 인덱스 튜닝, 쿼리 최적화, 읽기 전용 복제본(Read Replica), 캐싱 등 모든 수단을 먼저 동원하라. 샤딩은 이 모든 노력으로도 감당이 안 될 때 선택하는 길이다.
2. **분산 조인의 금지**: 샤드 간 조인은 성능의 재앙이다. 필요한 경우 데이터를 중복 저장(Denormalization)해서라도 각 샤드가 독립적으로 쿼리를 수행할 수 있는 구조를 만들어야 한다.
3. **추상화 레이어**: 애플리케이션 코드가 샤딩 로직에 오염되지 않도록 **Vitess**나 **ShardingSphere** 같은 미들웨어를 도입하여 '논리적 단일 DB'처럼 보이게 설계하라.

데이터는 중력을 가진 자산이다. 샤딩은 그 중력을 쪼개어 시스템을 자유롭게 만드는 기술이지만, 그 대가로 엔지니어의 정교한 관리 역량을 요구한다.
