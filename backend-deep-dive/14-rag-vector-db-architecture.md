# [백엔드 14편] RAG 아키텍처와 Vector DB — LLM 기반 서비스에서 백엔드 엔지니어는 무엇에 집중해야 하는가?

&nbsp;

ChatGPT의 등장은 전 세계를 열광시켰지만, 기업의 비즈니스 현장에서는 곧바로 차가운 현실에 직면했다. LLM(대형 언어 모델)이 내뱉는 그럴듯한 거짓말, 즉 **환각(Hallucination)** 현상과 모델 학습 시점 이후의 정보를 알지 못하는 **최신성 결여** 문제였다. 이를 해결하기 위해 모델을 새로 학습(Fine-tuning)시키는 것은 천문학적인 비용과 시간이 든다. 

&nbsp;

여기서 백엔드 엔지니어의 구원자로 등장한 것이 바로 **RAG(Retrieval-Augmented Generation)**다. 모델에게 지식을 가르치는 대신, 답변을 할 때 참고할 '참고서'를 건네주는 이 아키텍처는 이제 AI 서비스 백엔드의 표준이 되었다.

&nbsp;

# 1. RAG의 본질: 모델의 '기능'과 '지식'의 분리

&nbsp;

RAG는 LLM을 정보를 저장하는 '백과사전'이 아닌, 주어진 맥락을 이해하고 문장을 생성하는 **'추론 엔진'**으로만 사용한다. 

사용자의 질문이 들어오면 백엔드 시스템은 다음과 같은 과정을 거친다.
1. **검색(Retrieval)**: 질문과 관련된 최신 정보나 사내 문서를 외부 저장소에서 찾아낸다.
2. **증강(Augmentation)**: 찾아낸 정보를 원래 질문과 결합하여 LLM에게 전달할 '프롬프트'를 만든다.
3. **생성(Generation)**: LLM은 오직 제공된 정보만을 바탕으로 답변을 생성한다.

이 구조의 최대 장점은 데이터가 바뀌어도 모델을 재학습시킬 필요가 없으며, 답변의 근거를 사용자에게 제시할 수 있다는 점이다.

# 2. Vector DB: 의미 기반 검색(Semantic Search)의 메커니즘

&nbsp;

RAG 파이프라인에서 가장 중요한 구성 요소는 수천만 개의 문서 중에서 관련 있는 것을 순식간에 찾아내는 **Vector DB**다. 

&nbsp;

전통적인 키워드 검색(SQL `LIKE` 등)은 단어가 정확히 일치해야 하지만, Vector DB는 **의미적 유사도**를 계산한다.
- **임베딩(Embedding)**: 텍스트를 고차원 벡터 공간의 좌표로 변환한다. "강아지"와 "개"는 서로 다른 단어지만 벡터 공간에서는 매우 가까운 거리에 위치하게 된다.
- **유사도 연산**: 사용자의 질문이 들어오면 이를 벡터화한 뒤, Vector DB에서 '코사인 유사도' 등이 가장 높은 문서를 밀리초 단위로 찾아낸다.

# 3. 실전 엔지니어링 교훈: 시니어 엔지니어가 마주할 도전 과제

&nbsp;

많은 이들이 AI 모델의 파라미터 수에 집착할 때, 시니어 백엔드 엔지니어는 **데이터 파이프라인의 정합성**에 집중해야 한다.

1. **데이터 동기화 (ETL Pipeline)**: 원본 문서(Notion, Jira, PDF 등)가 수정되었을 때, 이를 즉시 임베딩하여 Vector DB에 반영하는 파이프라인의 지연 시간은 어떻게 관리할 것인가?
2. **검색 품질 최적화**: 단순히 벡터 검색만으로는 한계가 있다. 키워드 검색과 벡터 검색을 섞는 **하이브리드 검색(Hybrid Search)**과 검색 결과의 우선순위를 다시 매기는 **Re-ranking** 기술이 실제 서비스 품질을 결정한다.
3. **보안 및 권한 제어**: AI가 답변할 때 사용자가 접근 권한이 없는 문서를 참고하지 않도록 하는 필터링 레이어는 아키텍처의 필수 요소다.

RAG 아키텍처는 결국 AI 기술을 감싸는 정교한 **데이터 엔지니어링**의 산물이다. 모델보다 데이터의 흐름을 장악하는 자가 최상의 AI 서비스를 만든다.
