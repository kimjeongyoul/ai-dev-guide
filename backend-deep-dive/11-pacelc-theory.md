# [백엔드 11편] PACELC 이론 — 분산 환경에서 '데이터 일관성'과 '가용성'의 트레이드오프를 어떻게 넘어설 것인가?

&nbsp;

분산 시스템 설계자들에게 CAP 이론은 오랫동안 '거역할 수 없는 물리 법칙'처럼 여겨졌다. "일관성(C), 가용성(A), 네트워크 파티션 허용(P) 중 단 두 가지만 선택할 수 있다"는 이 명제는 명쾌했지만, 한편으로는 위험한 단순화였다. 실제 운영 환경에서 설계자를 괴롭히는 것은 네트워크가 끊겼을 때의 결단보다, 시스템이 멀쩡하게 돌아가는 99.9%의 시간 동안 발생하는 **지연 시간(Latency)**과 **정합성** 사이의 소리 없는 전쟁이기 때문이다. 

&nbsp;

본 글에서는 CAP 이론이 놓쳤던 '정상 상황의 Else'를 파고드는 **PACELC 이론**을 통해, 대규모 분산 환경에서 지연 시간을 최적화하기 위해 설계자가 무엇을 포기하고 무엇을 얻어야 하는지 심층 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. CAP 이론의 맹점: 네트워크는 늘 정상이다

&nbsp;

CAP 이론은 오직 네트워크 파티션(P) 상황만을 가정한다. 즉, "서버 간 통신이 끊겼을 때 서비스를 포기할 것인가(Consistency), 아니면 정합성을 포기하고 응답을 줄 것인가(Availability)"를 묻는다. 하지만 현실에서 네트워크 파티션은 예외적인 상황이다. 

&nbsp;

엔지니어가 매일 마주하는 진짜 문제는 **"네트워크가 정상일 때(Else), 사용자에게 10ms 안에 응답을 주기 위해 다른 노드로의 데이터 복제를 기다릴 것인가(Consistency), 아니면 지연 시간을 줄이기 위해 일단 응답할 것인가(Latency)"**이다. PACELC 이론은 바로 이 지점에서 CAP 이론의 빈틈을 메운다.

&nbsp;

&nbsp;

&nbsp;

# 2. PACELC의 8자 설계: P-A-C와 E-L-C의 교차점

&nbsp;

PACELC는 시스템의 상태를 두 가지 시나리오로 나눈다.

&nbsp;

## 2-1. [P] 상황: 장애 시의 생존 전략 (A vs C)
네트워크 파티션이 발생했을 때, 시스템은 가용성(A)을 택해 서비스 중단을 막을 것인지, 아니면 강한 일관성(C)을 유지하기 위해 서비스를 일시 중지할 것인지를 결정해야 한다. 
- **PA(Partition-Availability)**: 장애가 나도 응답은 한다. 데이터는 좀 틀릴 수 있다.
- **PC(Partition-Consistency)**: 장애가 나면 정합성을 위해 응답을 거부한다.

&nbsp;

## 2-2. [E] 상황: 정상 시의 성능 전략 (L vs C)
네트워크가 정상(Else)일 때의 트레이드오프가 핵심이다.
- **EL(Else-Latency)**: 응답 속도가 최우선이다. 복제는 비동기로 처리하며, 사용자는 일시적으로 과거 데이터를 볼 수 있다. (Eventual Consistency)
- **EC(Else-Consistency)**: 응답이 늦더라도 모든 노드에 데이터가 복제될 때까지 기다린다. (Strong Consistency)

# 3. 실전 엔지니어링 교훈: 시니어 설계자의 선택

&nbsp;

주니어 설계자는 종종 "데이터는 당연히 정확해야 하니 강한 일관성(EC)이 답이다"라고 말한다. 하지만 시니어 설계자는 **지연 시간(Latency)이 곧 비용이자 사용자 경험**임을 안다.

1. **PA/EL 모델 (DynamoDB, Cassandra)**: 전 세계를 대상으로 하는 소셜 서비스에 적합하다. 친구의 게시글이 0.5초 늦게 보인다고 해서 사용자가 이탈하지는 않지만, 페이지 로딩이 2초가 넘어가면 사용자는 떠난다.
2. **PC/EC 모델 (금융 시스템, Spanner)**: 단 1원의 오차도 허용되지 않는 결제 시스템의 숙명이다. 이 경우 인프라를 수직 확장하거나 전용 회선을 깔아서라도 물리적인 Latency를 억지로 끌어내려야 한다.

결국 PACELC 이론은 우리에게 질문을 던진다. **"당신의 비즈니스 도메인에서 '지연'과 '오류' 중 무엇이 더 치명적인가?"** 이 질문에 수치로 답할 수 있는 능력이 바로 분산 시스템 설계의 핵심이다.
