# [백엔드 12편] Write-Ahead Logging(WAL) — 왜 모든 고성능 데이터베이스는 실제 데이터보다 로그를 먼저 기록하는가?

&nbsp;

데이터베이스 엔지니어링의 세계에서 '성능'과 '안전'은 물과 기름 같은 관계다. 데이터를 디스크에 안전하게 기록하려 할수록 시스템은 느려지고, 속도를 높이려 하면 정전 한 번에 공든 탑이 무너진다. 하지만 PostgreSQL, MySQL, 그리고 전설적인 Kafka까지, 모든 고성능 데이터 플랫폼은 이 모순을 해결하기 위해 단 하나의 원칙을 고수한다. 바로 **"진짜 데이터를 쓰기 전에, 무엇을 할 것인지 로그부터 남겨라"**는 것이다.

&nbsp;

본 글에서는 디스크 I/O의 물리적 한계를 극복하고 장애 복구의 기적을 일궈낸 **WAL(Write-Ahead Logging)**의 내부 메커니즘을 심층 분석한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. Random Write의 지옥: 왜 DB는 쓰기 작업에 허덕이는가?

&nbsp;

현대의 SSD가 비약적으로 발전했음에도 불구하고, 디스크의 가장 치명적인 약점은 **Random Access**다. 데이터베이스의 실제 데이터 파일(B-Tree 등)은 데이터가 추가될 때마다 디스크의 이곳저곳을 찾아다니며 페이지를 수정해야 한다. 

&nbsp;

1,000건의 주문이 동시에 들어오면 DB는 디스크 헤더를 수천 번 움직여야 한다. 이 과정에서 발생하는 커널의 I/O Wait와 Context Switch는 CPU를 지치게 만들고, 전체 시스템의 응답 속도를 수직 하락시킨다. 여기서 엔지니어는 질문을 던진다. "어차피 기록해야 할 데이터라면, 일단 가장 빠른 방식으로 적어두고 나중에 정리하면 안 될까?"

&nbsp;

&nbsp;

---

&nbsp;

# 2. WAL의 해법: Sequential Write의 승리

&nbsp;

WAL은 데이터를 실제 위치에 쓰기 전에, 변경 내역을 별도의 **Append-only 로그 파일**에 기록한다. 이 방식이 압도적으로 빠른 이유는 단 하나, **Sequential Write(순차 쓰기)** 때문이다.

&nbsp;

1. **메모리(Buffer Pool) 우선 업데이트**: 데이터 변경 요청이 오면 먼저 메모리 상의 페이지를 수정한다. (Dirty Page 생성)
2. **WAL 로그 기록**: 변경된 내용(Redo Log)을 로그 파일 끝에 붙인다. 디스크 헤더가 움직일 필요 없이 끝에 이어 쓰기만 하면 되므로 물리적으로 매우 빠르다.
3. **성공 응답**: 로그가 디스크에 안전하게 기록(fsync)되는 즉시 사용자에게 성공을 알린다.
4. **체크포인트(Check-pointing)**: 시스템이 한가하거나 메모리가 찰 때, 비로소 Dirty Page들을 실제 데이터 파일로 옮겨 적는다.

&nbsp;

이 메커니즘 덕분에 DB는 사용자에게 "기록 완료"라고 말하면서도, 실제 무거운 쓰기 작업은 백그라운드에서 여유롭게 처리할 수 있게 된다.

&nbsp;

&nbsp;

&nbsp;

# 3. 실전 엔지니어링 교훈: 불변성(Immutability)과 복구의 미학

&nbsp;

WAL은 단순히 성능 도구가 아니다. 시스템이 정전이나 커널 패닉으로 갑자기 중단되었을 때, WAL은 **'진실의 원천'**이 된다.

- **Redo**: 메모리에는 있었지만 디스크(Data File)에는 반영되지 않은 변경 사항을 로그를 보고 다시 실행한다.
- **Undo**: 트랜잭션 도중 시스템이 멈췄다면, 로그의 이전 상태를 보고 데이터를 원상복구한다.

시니어 엔지니어라면 WAL의 원리에서 **"불변 로그(Immutable Log)가 상태(State)를 결정한다"**는 철학을 배워야 한다. Kafka가 초당 수백만 메시지를 처리할 수 있는 이유도, 결국 이 WAL의 순차 쓰기 원리를 극단적으로 활용해 메시지 큐 자체가 하나의 거대한 로그 파일처럼 동작하게 만들었기 때문이다. 성능을 원한다면 상태를 직접 바꾸려 하지 말고, 변화를 기록하는 로그에 집중하라.
