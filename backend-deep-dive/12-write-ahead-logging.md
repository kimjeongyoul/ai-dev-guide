# [백엔드 12편] Write-Ahead Logging(WAL) — 왜 모든 고성능 데이터베이스는 실제 데이터보다 로그를 먼저 기록하는가?

&nbsp;

PostgreSQL, MySQL, 그리고 Kafka까지. 현대 데이터 플랫폼의 정합성과 성능을 지탱하는 기술의 뿌리에는 **WAL(Write-Ahead Logging)**이 있습니다. 데이터를 직접 쓰는 것보다 로그를 남기는 것이 왜 더 빠르고 안전한지 그 비밀을 파헤칩니다.

&nbsp;

---

&nbsp;

# 1. WAL의 핵심: Random Write를 Sequential Write로 전환하라

&nbsp;

데이터베이스의 실제 데이터 파일(B-Tree 등)은 디스크의 이곳저곳에 흩어져 있습니다. 이를 직접 수정하는 것은 물리적 헤더가 바쁘게 움직여야 하는 **Random Write**를 유발하며, 이는 성능의 치명적인 병목이 됩니다.

&nbsp;

WAL은 이를 다음과 같이 해결합니다.
1. 모든 변경 사항을 메모리(Buffer Pool)에 반영하기 전, **WAL 로그 파일 끝에 순차적으로 기록**합니다.
2. 로그 파일 끝에 데이터를 붙이는 **Sequential Write**는 디스크 연산 중 가장 빠릅니다.
3. 실제 무거운 데이터 파일 업데이트(Check-pointing)는 백그라운드에서 여유 있게 수행합니다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. 성능 최적화와 장애 복구의 두 마리 토끼

&nbsp;

## 2-1. 쓰기 지연 제거를 통한 처리량 향상
사용자의 요청에 대해 로그 기록만 완료되면 즉시 성공 응답을 보낼 수 있습니다. 이는 전체 시스템의 응답 시간을 획기적으로 단축시키며, 디스크 I/O가 몰리는 상황에서도 안정적인 처리를 가능케 합니다.

## 2-2. 완벽한 데이터 재현 (Redo)
시스템이 갑자기 셧다운되어 메모리의 변경 사항이 날아가더라도, 디스크에 남은 WAL 로그가 '진실의 원천'이 됩니다. 재부팅 시 로그를 순차적으로 실행(Redo)하여 장애 발생 직전의 상태로 완벽하게 복구할 수 있습니다.

&nbsp;

&nbsp;

&nbsp;

# 3. 실전 엔지니어링 교훈: 불변성(Immutability)의 가치

&nbsp;

WAL이 성공한 이유는 로그 파일이 **'한 번 쓰면 변하지 않는(Append-only)'** 특성을 가졌기 때문입니다. 데이터의 불변성을 유지하며 순차적으로 쌓아나가는 방식은 Kafka와 같은 현대적인 이벤트 스트리밍 플랫폼의 근간이 되었습니다. 고성능 시스템을 설계한다면, 무거운 상태 변경보다 가벼운 로그 기록을 먼저 고민해야 합니다.
