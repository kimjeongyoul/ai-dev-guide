# [백엔드 7편] 합의 알고리즘과 쿼럼(Quorum) 설계 — 분산 시스템의 신뢰성을 지탱하는 합의의 원리는 무엇인가?

&nbsp;

"누가 대장인가?" 수천 대의 서버가 연결된 분산 시스템에서 가장 어려운 질문은 이것이다. 네트워크는 언제든 끊길 수 있고, 특정 서버는 예고 없이 죽는다. 만약 서버들이 서로 자신이 리더라고 주장하며 데이터를 쓰기 시작한다면 정합성은 순식간에 파괴된다. 이른바 **Split Brain** 현상이다. 

&nbsp;

분산 환경에서 물리적으로 떨어진 노드들이 단 하나의 진실에 도달하기 위해 사용하는 마법이 바로 **합의 알고리즘(Consensus Algorithm)**이다. 본 글에서는 현대 분산 시스템의 근간인 Raft와 Paxos, 그리고 그 기저에 흐르는 **쿼럼(Quorum)**의 원리를 해부한다.

&nbsp;

&nbsp;

---

&nbsp;

# 1. 쿼럼(Quorum) 설계: 과반수의 마법

&nbsp;

합의의 핵심은 "전체 노드가 다 살아있지 않아도 결정할 수 있는가?"이다. 쿼럼(Quorum)은 의사결정에 필요한 **최소한의 유효 투표 수**를 의미한다. 

&nbsp;

보통 `(N/2 + 1)`의 공식을 사용한다. 예를 들어 5대의 서버 중 3대만 동의하면, 나머지 2대가 네트워크에서 고립되더라도 그 2대는 결코 과반수를 만들 수 없다. 이 단순한 수학적 원리가 분산 시스템에서 데이터가 충돌하는 것을 방지하는 강력한 가드레일이 된다.

&nbsp;

&nbsp;

---

&nbsp;

# 2. Raft 알고리즘: 이해할 수 있는 합의의 표준

&nbsp;

과거 Paxos 알고리즘은 논문 자체가 난해하여 구현이 매우 어려웠다. 이를 해결하기 위해 등장한 것이 **Raft**다. 

&nbsp;

1. **리더 선출 (Leader Election)**: 리더가 죽으면 노드들은 '임기(Term)'를 늘리고 투표를 시작한다. 쿼럼을 얻은 노드가 새로운 리더가 된다.
2. **로그 복제 (Log Replication)**: 모든 쓰기 요청은 리더에게 간다. 리더는 로그를 남기고 다른 노드들에게 전파한다. 과반수 이상의 노드가 로그를 기록했음을 확인하면(Commit), 비로소 데이터를 확정한다.

Zookeeper, etcd, MongoDB 등 수많은 고성능 분산 플랫폼이 이 원리를 통해 고가용성과 정합성을 유지한다.

&nbsp;

&nbsp;

&nbsp;

# 3. 실전 엔지니어링 교훈: 네트워크 파티션의 숙명

&nbsp;

시니어 설계자는 합의 알고리즘을 도입할 때 다음의 트레이드오프를 정밀하게 계산해야 한다.

1. **지연 시간과 노드 수**: 노드 수가 많아질수록 가용성은 높아지지만, 합의를 위한 통신 오버헤드(네트워크 홉)가 늘어나 응답 속도가 느려진다. 보통 3대나 5대의 홀수 구성이 최적으로 꼽히는 이유다.
2. **CAP 이론과의 접점**: 합의 알고리즘 기반 시스템은 기본적으로 **CP(Consistency & Partition Tolerance)** 지향적이다. 과반수 이상의 노드가 죽으면 시스템은 가용성(A)을 포기하고 모든 응답을 중단한다. 
3. **읽기 성능 최적화**: 모든 읽기를 리더에게 집중시키면 성능 병목이 생긴다. 정합성이 조금 떨어져도 된다면 팔로워(Follower)에게 읽기를 허용하는 설계를 병행해야 한다.

합의 알고리즘은 단순한 알고리즘을 넘어, **'불확실한 네트워크 위에서 확실한 진실을 세우는 철학'**이다. 분산 시스템 설계자의 수준은 이 합의의 비용을 얼마나 정확하게 이해하고 있느냐로 결정된다.
