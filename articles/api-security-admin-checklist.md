# 관리자 API를 안전하게 만드는 실전 체크리스트

이 문서는 백오피스 관리자 API를 운영하기 전에 빠르게 확인할 수 있는 실전 점검표다. 모든 항목을 한 번에 완벽하게 맞추는 것이 목적은 아니다. 우선순위가 높은 것부터 빠르게 누락 여부를 확인하는 데 목적이 있다.

## 1. 인증

- 모든 관리자 API가 로그인 세션 또는 토큰 검증 뒤에 있는가
- 프런트 화면 보호와 API 보호를 별도로 하고 있는가
- 테스트용 우회 로그인이나 mock 계정이 운영 코드에 남아 있지 않은가
- 관리자 세션 만료 정책이 있는가

## 2. 권한

- 관리자 역할이 최소 2개 이상으로 분리되어 있는가
- 역할별로 가능한 기능이 다르게 설정되어 있는가
- 배포, 계정 삭제, 권한 변경, 서비스 제어는 상위 권한만 가능한가
- 단순 로그인 여부만으로 모든 기능이 열리지 않는가

## 3. 객체 접근 제어

- 특정 멤버, 점포, 키오스크, 패치에 대한 접근 가능 범위를 검사하는가
- URL의 `id`만 알면 아무 객체나 조회 또는 수정 가능한 구조가 아닌가
- 관리자라도 담당 범위를 넘는 데이터에는 접근하지 못하는가

## 4. 입력 검증

- POST, PUT, PATCH 요청 body를 스키마로 검증하는가
- 필수값, 길이, enum, 형식을 확인하는가
- 외부 시스템으로 넘기기 전에 입력을 한 번 더 정리하는가
- 파일 업로드 시 확장자만 아니라 크기와 저장 경로도 제한하는가

## 5. 화이트리스트 업데이트

- 수정 API가 요청 body 전체를 그대로 update 하지 않는가
- 수정 가능한 필드만 명시적으로 골라서 반영하는가
- 시스템 필드나 권한 필드가 클라이언트 요청으로 덮어써지지 않는가

## 6. 민감 기능 보호

- 배포 실행 API는 일반 관리자에게 열려 있지 않은가
- 계정 삭제와 권한 변경에는 추가 확인 절차가 있는가
- 서비스 재시작, 장비 제어, 환경 설정 변경은 별도 보호가 있는가
- 정말 위험한 기능은 재인증 또는 확인 문자열 입력을 요구하는가

## 7. 감사 로그

- 누가 무엇을 변경했는지 기록하는가
- 로그인 실패, 권한 거부, 배포 실행, 설정 변경이 로그에 남는가
- 로그에 actor, action, target, timestamp, ip 정도는 남는가
- 감사 로그가 일반 애플리케이션 로그와 구분되는가

## 8. Rate Limit과 남용 방지

- 로그인 API에 rate limit이 있는가
- 업로드 API에 횟수 또는 크기 제한이 있는가
- 명령 발행 API에 반복 호출 방지가 있는가
- 대량 조회 API에 페이지 제한이 있는가

## 9. 네트워크와 운영 설정

- 관리자 API는 외부 공개 없이 VPN 또는 내부망 뒤에 있는가
- 장비 제어 API는 허용된 IP만 접근 가능한가
- 운영과 개발 환경이 분리되어 있는가
- 운영 환경에서 mock, debug, test 엔드포인트가 제거되었는가

## 10. 외부 연동

- 외부 API 응답을 그대로 신뢰하지 않는가
- 응답 구조와 상태값을 검증하는가
- 임의 URL 호출이 가능한 구조가 아닌가
- 외부 연동 실패 시 내부 상태를 안전하게 처리하는가

## 최소 권장 기준

운영에 들어가기 전 최소한 아래는 맞추는 것이 좋다.

- 관리자 API 인증 적용
- 역할 기반 권한 분리
- 민감 기능 보호
- 입력 검증
- 화이트리스트 업데이트
- 감사 로그
- mock 계정과 테스트 엔드포인트 제거

## 납품 직전 우선순위

납품 직전에는 구조 개편보다 다음을 우선 본다.

1. 인증 없는 관리자 API 제거
2. 민감 기능 권한 점검
3. mock 또는 우회 코드 제거
4. 업로드와 배포 API 접근 제한
5. 배포 전 실제 운영 경로와 환경값 재확인

## 한 줄 기준

관리자 API는 내부 시스템이 아니라 운영 핵심 시스템이다. 조회 API처럼 다루지 말고, 변경과 제어가 가능한 보안 민감 API로 취급해야 한다.
